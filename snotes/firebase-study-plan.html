Practical tips for the Spark plan
Use Firestore rules as your backend
Without Cloud Functions, Firestore Security Rules are your primary access control layer. Master request.auth, resource.data, and get() lookups — they replace most server-side validation.
Cache reads aggressively
Firestore's 50K reads/day limit is generous but finite. Use onSnapshot listeners instead of repeated getDocs calls — the listener only counts reads on initial load and real changes.
Denormalise your data
Firestore charges per document read, not per query. Flatten data structures so one read gives you everything you need. Avoid patterns that require joining multiple collections for a single screen.
Firebase Emulator Suite is essential
Run Auth, Firestore, Storage, and Hosting locally with firebase emulators:start. Emulators don't count against your quota — do all development and testing locally before deploying.
Monitor usage in the console
Check the Firebase console's Usage tab daily during learning. Firestore read counts can spike unexpectedly if you have a listener on a large collection with frequent writes.
Blaze is still free-ish
When you're ready for Cloud Functions, upgrading to Blaze doesn't mean paying — the free-tier quotas still apply. You only pay when you exceed them. Set a budget alert at $1.
