Object-Oriented JavaScript

JavaScript can absolutely be written in an object-oriented style, but it does not work exactly like Java or Python. Under the hood, JavaScript is prototype-based: objects can inherit behavior directly from other objects, and the modern class syntax is mostly a cleaner layer on top of prototypes. In practice, "doing OOP in JavaScript" usually means creating objects that hold state, attaching methods that act on that state, hiding implementation details, and composing behavior carefully instead of building deep inheritance trees.

The Core Idea

Think in terms of objects with data plus behavior, but remember that JavaScript favors flexibility over rigid class systems.

const account = {
  owner: 'Mina',
  balance: 0,
  deposit(amount) {
    this.balance += amount;
  },
  withdraw(amount) {
    if (amount > this.balance) {
      throw new Error('Insufficient funds');
    }
    this.balance -= amount;
  }
};

account.deposit(50);
account.withdraw(20);
console.log(account.balance); // 30

Classes: The Usual Starting Point

If you want something that feels closest to OOP in Java or Python, use class. It is the most readable starting point for teams.

class User {
  constructor(name, role = 'member') {
    this.name = name;
    this.role = role;
  }

  describe() {
    return `${this.name} has role ${this.role}`;
  }
}

const user = new User('Aiko');
console.log(user.describe()); // 'Aiko has role member'

const admin = new User('Ren', 'admin');
console.log(admin.describe()); // 'Ren has role admin'

constructor creates the instance. Methods live on the prototype, so they are shared instead of copied onto every object, and the default role is used only when you do not pass a second argument.

What Classes Really Use: Prototypes

JavaScript inheritance is based on prototype chains. Classes are syntax sugar over that mechanism.

function Animal(name) {
  this.name = name;
}

Animal.prototype.speak = function () {
  return `${this.name} makes a sound`;
};

const cat = new Animal('Mochi');
console.log(cat.speak()); // 'Mochi makes a sound'

You do not need to write code this way often anymore, but understanding prototypes explains how class, method lookup, and inheritance actually work.

Encapsulation

JavaScript does not force strict access modifiers like Java, but you still have solid ways to hide internal details.

class Counter {
  #count = 0;

  increment() {
    this.#count += 1;
  }

  get value() {
    return this.#count;
  }
}

const counter = new Counter();
counter.increment();
console.log(counter.value); // 1
// counter.#count; // SyntaxError

Use private fields like #count when you want hard encapsulation in modern JavaScript.

Inheritance: Use It, But Lightly

Inheritance works, but deep hierarchies become brittle fast. Prefer it for clear "is-a" relationships.

class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  speak() {
    return `${this.name} barks`;
  }
}

const dog = new Dog('Cookie');
console.log(dog.speak()); // 'Cookie barks'

extends sets up the prototype chain, and child classes can override parent behavior.

Composition: Usually the Better Default

In JavaScript, composition is often cleaner than inheritance. Build objects by combining small behaviors.

const canDrive = (state) => ({
  drive() {
    return `${state.name} is driving`;
  }
});

const canCharge = (state) => ({
  charge() {
    state.battery = 100;
    return `${state.name} is fully charged`;
  }
});

function createElectricCar(name) {
  const state = { name, battery: 20 };
  return {
    ...canDrive(state),
    ...canCharge(state),
    getBattery() {
      return state.battery;
    }
  };
}

const car = createElectricCar('Volt');
console.log(car.drive());
console.log(car.charge());

This pattern is common in JavaScript because it avoids rigid inheritance trees and makes behavior easier to reuse.

Factories vs Classes

You do not have to use new. Factory functions are a very JavaScript-native OOP approach.

function createTimer(label) {
  let seconds = 0;

  return {
    tick() {
      seconds += 1;
    },
    read() {
      return `${label}: ${seconds}s`;
    }
  };
}

const timer = createTimer('Workout');
timer.tick();
console.log(timer.read()); // 'Workout: 1s'

Factories pair nicely with closures when you want private state without classes.

TypeScript Makes OOP Safer

JavaScript itself stays dynamic, but TypeScript helps document object shapes and method contracts.

interface Shape {
  area(): number;
}

class Rectangle implements Shape {
  constructor(private width: number, private height: number) {}

  area(): number {
    return this.width * this.height;
  }
}

const shape: Shape = new Rectangle(4, 5);
console.log(shape.area()); // 20

TypeScript does not change the runtime model, but it does make OOP-heavy codebases easier to reason about.

Practical Advice

How to "do OOP" in JavaScript in real projects:

// Good defaults
// 1. Start with plain objects for simple data.
// 2. Use classes when instances share a clear lifecycle and behavior.
// 3. Prefer composition over deep inheritance.
// 4. Use private fields or closures for hidden state.
// 5. Keep methods small and focused on the object's responsibility.

JavaScript is not missing OOP; it just offers a looser, more prototype-driven version of it. The best approach is usually lightweight objects, small classes where they help, and composition whenever inheritance starts to feel forced.
