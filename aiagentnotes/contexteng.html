Context Engineering for AI Applications

The discipline of deciding what information to provide, what to discard, and how to maintain state within a limited context window — so an LLM consistently generates the highest-quality responses.

What is Context Engineering? Definition

Unlike prompt engineering — which focuses primarily on textual instructions — context engineering manages the entire scope of input information fed to a model.

This includes system prompts, conversation history, tool definitions, retrieval results from external databases (RAG), and multimodal data such as images or audio — all orchestrated to fit within the model's token budget while preserving the information that matters most.

Why Context Engineering Matters

In modern AI development — especially when building complex agents — context engineering directly determines the quality of the user experience.

Physical Constraints

LLMs have a finite context window — the total tokens (input + output) they can process at once. Exceeding it causes the model to forget earlier information, breaking multi-turn dialogues and large-scale tasks like repository analysis.

Context Rot

Even within the token limit, models suffer from Context Rot — a phenomenon where comprehension degrades as input length grows. Performance in complex tasks can decline significantly after only a few thousand tokens.

Cost & Latency

AI APIs are billed per token, and larger contexts increase latency. Techniques like Context Caching store fixed parts of the context — such as system prompts or RAG documents — drastically reducing both cost and response time on repeated calls.

Key Domains of Context Engineering

Acquisition  & Generation

Designing system prompts, searching external knowledge bases (RAG), and dynamically assembling context that is specific to each task and user.

Processing

Handling long-context inputs efficiently and using self-reflection or correction loops to iteratively improve output quality before returning results to the user.

Management

Designing memory hierarchies (short-term vs. long-term), compressing information via summarization, and orchestrating context windows across multiple specialized agents.

Practical Strategies for Developers

Sub-agent Architecture

Prevent "memory pollution" by delegating specific tasks to specialized sub-agents that each have their own independent context window, keeping every agent's context lean and focused.

Progressive Disclosure

Instead of front-loading all rules, use "Skills" to inject task-specific instructions and templates only when the LLM determines they are needed — keeping the baseline context compact.

Structured Note-taking

Maintain long-term memory by having the AI record important decisions and state to external files, which are then re-read selectively in future sessions rather than kept in the live context.

Context Budget Decisions Practical Guide

Context engineering is not just "add more context" — it is deciding what belongs in the model input right now. Every token in the context window competes for attention and budget:

Instructions + user goal + selected memory + retrieved facts + tool definitions + output format = model input

1

Stable instructions: system prompt, role, policy, output format.

2

User-specific state: preferences, active task, recent decisions.

3

Retrieved facts: only the most relevant chunks, preferably with source IDs.

4

Tool definitions: only expose tools the agent may need for this task.

5

Scratchpad or plan: keep it compact and remove stale steps.

6

Output budget: leave enough tokens for the final answer.

Context Caching in Practice Python Anthropic SDK

Cache a large knowledge base or system prompt so it is only tokenised once. Subsequent calls read from cache at a significantly reduced input cost, directly lowering both spend and latency. See Anthropic prompt caching docs for current pricing, which changes over time.

context_cache.py 
# Requires: pip install anthropic
# Set your key: export ANTHROPIC_API_KEY="sk-ant-..."
import anthropic

client = anthropic.Anthropic()

# The large knowledge base is cached after the first call.
# Re-use is billed at ~10% of normal input token price.
response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": open("knowledge_base.txt").read(),
            "cache_control": {"type": "ephemeral"}  # mark for caching
        }
    ],
    messages=[
        {"role": "user", "content": "Summarize the key findings."}
    ]
)

print(response.content[0].text)

# Inspect token usage to verify cache hits
usage = response.usage
print(f"Input tokens:       {usage.input_tokens}")
print(f"Cache write tokens: {usage.cache_creation_input_tokens}")
print(f"Cache read tokens:  {usage.cache_read_input_tokens}")

On the first call, cache_creation_input_tokens will be non-zero. On every subsequent call with the same content, cache_read_input_tokens reflects the tokens served from cache — the fastest way to validate your caching strategy is working.
