What Other MCP Servers Help Write Tests?

Playwright MCP covers the browser. Writing good test cases — not just browser actions — needs a few more tool servers: one to read and write files in the repo, one to check history and open PRs, one to hit APIs directly, and one to remember selectors across runs.

Servers Worth Adding Overview

Server Tools Why the agent needs it
filesystem read_file, write_file, list_directory Read the existing tests/ tree so the agent reuses page objects instead of duplicating them, then write the new spec file.
git git_diff, git_log, git_status See what recently changed in the site's source so new tests target actual recent UI changes, not stale assumptions.
GitHub create_pull_request, get_issue, list_workflow_runs Open a PR with the generated test, link it to the issue that requested it, and check whether CI is green.
fetch fetch Hit an endpoint directly for API-level assertions (status code, JSON shape) that don't need a full browser.
memory store, recall Persist selectors and known-flaky tests across sessions instead of re-discovering them every run — see Context Engineering.
a11y-scan (custom) run_axe Wrap axe-core as a tool so the agent can write accessibility assertions, not just visual ones.

Building a Custom "Coverage" Server TypeScript SDK

Not everything has an off-the-shelf server. Here's a small custom one that compares the site's routes against the existing spec files, so a "what should I test next?" prompt gets a real answer instead of a guess.

coverage-server.ts 
// npm i @modelcontextprotocol/sdk zod
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import { z } from 'zod';
import { readdirSync, readFileSync } from 'fs';

const server = new McpServer({ name: 'test-coverage', version: '1.0.0' });

server.tool(
  'list_untested_routes',
  'Compare site routes against existing spec files and list ones with no test.',
  { routes: z.array(z.string()) },
  async ({ routes }) => {
    const specFiles = readdirSync('tests').filter((f) => f.endsWith('.spec.ts'));
    const testedText = specFiles
      .map((f) => readFileSync(`tests/${f}`, 'utf-8'))
      .join('\n');

    const untested = routes.filter((route) => !testedText.includes(route));
    return { content: [{ type: 'text', text: JSON.stringify(untested) }] };
  }
);

await server.connect(new StdioServerTransport());

Wiring Multiple Servers Together Config

A single config file lists every server the agent can reach. Adding the coverage server above later means adding one entry — the harness or MCP-native host discovers its tools automatically. 

{
  "mcpServers": {
    "playwright": { "command": "npx", "args": ["@playwright/mcp@latest", "--headless"] },
    "filesystem": { "command": "npx", "args": ["@modelcontextprotocol/server-filesystem", "./tests"] },
    "github": { "command": "npx", "args": ["@modelcontextprotocol/server-github"] },
    "coverage": { "command": "node", "args": ["coverage-server.js"] }
  }
}

Safety Checklist for These Servers Important

  • Scope the filesystem server to the tests/ directory only — never the whole repo.
  • Give the GitHub server a token that can open PRs but not merge or push to main directly.
  • Restrict fetch to the site's own domain(s) so the agent can't be prompted into calling arbitrary URLs.
  • Treat every generated PR as a draft for human review — none of this should auto-merge.
