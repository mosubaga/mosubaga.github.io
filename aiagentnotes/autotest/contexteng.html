Context Engineering for Test Agents

A raw browser_snapshot of a real page can be thousands of tokens of accessibility tree. Feed that in full, every call, across a growing test suite, and a small local model runs out of room to think — or starts skimming and hallucinating selectors. Context Engineering is what keeps test run #500 as reliable as test run #1.

The Problem: Snapshots Are Big, Context Windows Aren't Why It Matters

A local model has a smaller, more crowded context window than a hosted frontier model, and suffers from context rot — comprehension quietly degrading as input grows — sooner. An agent that dumps a full page snapshot, the whole existing test file, and a long failure history into every prompt will eventually spend most of its context re-reading things it already saw instead of fixing the one broken locator in front of it.

The Three Domains, Applied to Test Agents

Acquisition

Pull only the subtree of the accessibility snapshot relevant to the component under test — the <nav>, not the whole page — plus the one failing test's error, never the full CI log.

Processing

Subagent isolation (see Subagents) keeps the Fixer from ever seeing the Reviewer's rules, and lets the Flake Triager's rerun summary feed the Fixer only when it's a real regression — not on every flaky blip.

Management

Long-lived state — known-good selectors, known-flaky tests — lives outside the LLM entirely, in a small JSON or SQLite store, and only the relevant slice is ever loaded into a prompt.

Three Strategies That Do the Work

Per-Test Context Reset

Every spec file gets a fresh context. The Fixer never carries yesterday's failing test into today's — each run starts from the current error and the current snapshot only.

Snapshot Trimming

Filter the accessibility tree to the region around the element the test cares about, instead of passing the entire page every time a selector needs re-checking.

Selector Knowledge Base

Known-good locators for common elements (nav, footer, hero) are cached once and reused, so the Generator doesn't re-derive them from a full snapshot for every new spec file.

The Fixer's Context Budget Practical Guide

What actually goes into the Fixer's prompt for one failing test, in priority order:

Role & rules — the Fixer's fixed system prompt (never touch the assertion).

The one error message — locator timeout, expected vs. actual — not the full test log.

The failing spec file — just that file, not the whole tests/ tree.

A trimmed snapshot — the subtree around the last-known-good selector, capped by token budget.

Output format — return only the patched locator line, not a full file rewrite.

Trimming a Snapshot and Caching Selectors TypeScript

Two habits keep a run both correct and fast: capping the accessibility tree to a token budget instead of shipping the whole thing, and reading known selectors from a small cache before ever asking the model to re-derive them.

context-budget.ts 
import { readFileSync, writeFileSync, existsSync } from 'fs';

const MAX_SNAPSHOT_WORDS = 600; // keep the snapshot from crowding out the actual error
const CACHE_PATH = '.cache/selectors.json';

interface SelectorCache {
  [component: string]: string;
}

function loadSelectorCache(): SelectorCache {
  // Known-good locators, reused instead of re-derived from a full snapshot.
  return existsSync(CACHE_PATH) ? JSON.parse(readFileSync(CACHE_PATH, 'utf-8')) : {};
}

function saveSelector(component: string, locator: string): void {
  const cache = loadSelectorCache();
  cache[component] = locator;
  writeFileSync(CACHE_PATH, JSON.stringify(cache, null, 2));
}

function trimSnapshot(snapshot: string, maxWords = MAX_SNAPSHOT_WORDS): string {
  const words = snapshot.split(/\s+/);
  return words.length <= maxWords ? snapshot : words.slice(0, maxWords).join(' ') + ' …(truncated)';
}

function buildFixerPrompt(errorMessage: string, specFile: string, fullSnapshot: string): string {
  return (
    `Error:\n${errorMessage}\n\n` +
    `Spec file:\n${specFile}\n\n` +
    `Relevant snapshot:\n${trimSnapshot(fullSnapshot)}`
  );
}
