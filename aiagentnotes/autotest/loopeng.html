Can Loop Engineering Update Tests?

Yes — this is where the Fixer subagent stops being a one-shot call and becomes a bounded loop: run, inspect, patch, rerun, repeated until the suite is green or an iteration cap is hit. The engineering part isn't the loop itself, it's the guardrails that keep it from quietly rewriting a test's intent just to make it pass.

The Update-Test Loop Overview

npx playwright test runs the suite against mosubaga.github.io
Flake Triager reruns failures — real regressions continue, flaky ones stop here
Playwright MCP re-inspects the live page for the failing element
Fixer subagent patches the locator, Reviewer checks the assertion is untouched
Rerun that one test — pass exits the loop, fail increments the counter
Cap hit → open a draft PR with the diff and a summary, and stop — a human decides

A Bounded Fix Loop TypeScript

The cap is the whole point: an unbounded loop that keeps "trying again" is a bug, not a feature. This one gives up after three attempts and hands the diff to a human instead of looping forever.

fix_loop.ts 
import { runSubagent } from './subagents';
import { runSingleTest, type TestResult } from './test-runner';
import { openDraftPullRequest } from './github';

const MAX_ATTEMPTS = 3;

async function healTest(specFile: string, testTitle: string): Promise<void> {
  let lastResult: TestResult = await runSingleTest(specFile, testTitle);
  let attempt = 0;

  while (!lastResult.passed && attempt < MAX_ATTEMPTS) {
    attempt++;
    console.log(`Attempt ${attempt}/${MAX_ATTEMPTS} for "${testTitle}"`);

    const patch = await runSubagent(
      'fixer',
      `File: ${specFile}\nTest: ${testTitle}\nError: ${lastResult.error}`
    );

    const review = await runSubagent('reviewer', patch);
    if (review.includes('ASSERTION_CHANGED')) {
      throw new Error(`Fixer altered an assertion in ${specFile} — refusing to apply`);
    }

    await applyPatch(specFile, patch);
    lastResult = await runSingleTest(specFile, testTitle);
  }

  if (!lastResult.passed) {
    // Cap reached — stop looping and let a person look at the diff.
    await openDraftPullRequest({
      title: `Could not auto-heal: ${testTitle}`,
      body: `${attempt} attempts made. Last error:\n\n${lastResult.error}`,
    });
  }
}

Guardrails That Make This Safe Important

  • Cap iterations. Three tries and escalate, not thirty — an agent that "just needs one more attempt" forever is a runaway loop, not progress.
  • Assert intent, not output. The Reviewer rejects any patch that changes what's being asserted — the loop may only fix how an element is found, never what is expected of it.
  • No silent merges. Every outcome — healed or escalated — lands as a draft PR with a diff, never a direct commit to main.
  • Log every attempt. A three-attempt trail in the PR description tells a reviewer exactly what the agent tried and why it gave up.

Running It on a Schedule Automation

The loop doesn't have to wait for a failing CI run to trigger it — a nightly job catches drift on mosubaga.github.io before a human even notices the suite is red. 

# crontab -e — nightly at 2am, well before the morning's first CI run
0 2 * * * cd /opt/test-agent && npm run heal-suite >> logs/heal.log 2>&1
