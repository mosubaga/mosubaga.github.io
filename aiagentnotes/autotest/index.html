Let an Agent Write (and Fix) Your Playwright Tests

A walkthrough of using AI agents, Playwright MCP, and a local Ollama LLM to write, extend, and self-heal TypeScript Playwright tests — using https://mosubaga.github.io as the target site throughout.

Start with Playwright MCP See the Subagents

The Agent Loop

"Write a test that checks the nav on mosubaga.github.io"
Ollama calls Playwright MCP to open & inspect the live page
Agent emits a TypeScript *.spec.ts file
npx playwright test runs it in CI
On a failure, a loop re-inspects the page and heals the test

The Scenario

You maintain https://mosubaga.github.io and want AI to carry more of the test-writing and test-maintenance load. This site is about how, using:

1

Playwright MCP so an LLM can navigate, inspect, and screenshot the live site instead of guessing at markup.

2

A local Ollama model as the brain behind the agent — no cloud API required.

3

Focused subagents — a generator, a fixer, a bug hunter, a reviewer — instead of one do-everything prompt.

4

Context & loop engineering to keep the agent's prompts small and let it retry a fix until the suite is green.

Assumptions Constraints

  Local LLM via Ollama   Model already downloaded   TypeScript throughout   No cloud API required

Seven Questions, Seven Building Blocks

Each page below answers one question about building and running this kind of test agent.

Q1 · Setup

How do I set up Playwright MCP with a local LLM?

Install the MCP server, wire it to Ollama with a small tool-calling loop, and let the model write tests from a live page.

 Q2 · Protocol

What other MCP servers help write tests?

Filesystem, Git/GitHub, fetch, and memory servers round out what Playwright MCP alone can't do.

 Q3 · Delegation

What subagents can we build?

A generator, fixer, bug hunter, reviewer, and flake triager — each with a narrow job and toolset.

 Q4 · Memory

How does Context Engineering fit in?

Trimmed snapshots and a selector knowledge base keep a small local model focused per test file.

 Q5 · Iteration

Can Loop Engineering update tests?

Yes — a bounded run-inspect-patch-rerun loop with guardrails so it can't silently rewrite intent.

 Q6 · Ecosystem

What other AI agents can help?

Claude Code, Cursor, GitHub Copilot, and Codex CLI — how each fits the MCP-driven workflow.

 Q7 · Upkeep

How does AI maintain tests long-term?

Self-healing selectors, coverage gap-finding, and bug-hunting crawls of the live site.

Nice to Have: A Plain Playwright Test

This site isn't a Playwright tutorial, but here's the kind of TypeScript test the agents below read, write, and fix — a real spec against the live homepage.

tests/homepage.spec.ts 
import { test, expect } from '@playwright/test';

test.describe('mosubaga.github.io homepage', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('https://mosubaga.github.io');
  });

  test('loads with the expected title', async ({ page }) => {
    await expect(page).toHaveTitle(/./); // page has a non-empty <title>
  });

  test('primary navigation is visible', async ({ page }) => {
    const nav = page.locator('nav').first();
    await expect(nav).toBeVisible();
  });

  test('has no obviously broken images', async ({ page }) => {
    const images = page.locator('img');
    const count = await images.count();
    for (let i = 0; i < count; i++) {
      const naturalWidth = await images.nth(i).evaluate(
        (img: HTMLImageElement) => img.naturalWidth
      );
      expect(naturalWidth).toBeGreaterThan(0);
    }
  });
});

Tech Stack

LLM

Ollama running a locally downloaded tool-calling model (e.g. qwen2.5:7b) for planning and code generation.

Browser Tooling

@playwright/mcp exposes navigate/click/snapshot as MCP tools the model can call directly.

Test Runner

@playwright/test with TypeScript specs, run locally and in CI.

Automation

A scheduled or CI-triggered loop reruns the suite and hands failures to the fixer subagent.
