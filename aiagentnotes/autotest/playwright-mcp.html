Setting Up Playwright MCP on a Local LLM

Ollama doesn't speak MCP on its own — it's a chat/tool-calling API. Playwright MCP is a standalone server that exposes a real browser as MCP tools. The bridge between them is a small TypeScript harness: it lists Playwright MCP's tools, hands them to Ollama as function-calling tools, and forwards every tool call the model makes back to the browser.

What Playwright MCP Actually Gives You @playwright/mcp

Microsoft's Playwright MCP server drives a real Chromium/Firefox/WebKit instance and exposes it as callable tools — an accessibility-tree snapshot instead of raw HTML, so a model gets a compact, structured view of the page it can reason about, click into, and type into, without ever needing pixel-level vision.

Installing the Pieces

1 npm init -y && npm i -D @playwright/test @playwright/mcp typescript
2 npx playwright install — downloads the actual browser binaries
3 ollama pull qwen2.5:7b — a local model with tool-calling support
4 npm i -D @modelcontextprotocol/sdk ollama — the client libraries for the harness
5 Run the harness — it launches @playwright/mcp over stdio and starts chatting

Not every local model supports tool calling reliably — qwen2.5, qwen3, llama3.1, and mistral-nemo all do. Smaller quantizations of the same model family also work but call tools less reliably; test with a real prompt before trusting one in a loop.

Pointing an MCP Client at the Server Config

If you're driving Playwright MCP from an MCP-native host (Claude Code, Cursor, Claude Desktop) rather than a custom harness, this is the entire setup — one entry in its MCP config file.

mcp.json 
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest", "--headless"]
    }
  }
}

Wiring Ollama to Playwright MCP TypeScript

Since Ollama can't launch MCP servers itself, this harness does three things: start @playwright/mcp as a subprocess over stdio, translate its tool list into the schema Ollama's chat() expects, and loop — feed the model's tool calls back into MCP until it stops calling tools and returns a final answer.

agent.ts 
import { Client } from '@modelcontextprotocol/sdk/client/index.js';
import { StdioClientTransport } from '@modelcontextprotocol/sdk/client/stdio.js';
import ollama from 'ollama';

const transport = new StdioClientTransport({
  command: 'npx',
  args: ['@playwright/mcp@latest', '--headless'],
});

const mcp = new Client({ name: 'ollama-playwright-agent', version: '1.0.0' });
await mcp.connect(transport);

// Translate MCP tool definitions into the schema Ollama expects.
const { tools } = await mcp.listTools();
const ollamaTools = tools.map((t) => ({
  type: 'function' as const,
  function: { name: t.name, description: t.description, parameters: t.inputSchema },
}));

const messages = [
  {
    role: 'system',
    content:
      'You write Playwright TypeScript tests. Use the browser tools to inspect the ' +
      'real page before writing a locator — never guess selectors.',
  },
  {
    role: 'user',
    content:
      'Open https://mosubaga.github.io, snapshot the homepage, and write a ' +
      'Playwright test that asserts the primary nav is visible.',
  },
];

while (true) {
  const response = await ollama.chat({ model: 'qwen2.5:7b', messages, tools: ollamaTools });
  messages.push(response.message);

  if (!response.message.tool_calls?.length) {
    console.log(response.message.content); // final answer — the generated test
    break;
  }

  for (const call of response.message.tool_calls) {
    const result = await mcp.callTool({
      name: call.function.name,
      arguments: call.function.arguments,
    });
    messages.push({ role: 'tool', content: JSON.stringify(result.content) });
  }
}

Tools an Agent Actually Calls Reference

Tool Used for
browser_navigate Open https://mosubaga.github.io or a specific route in it.
browser_snapshot Get a structured accessibility tree — this is what the model actually reads to find locators.
browser_click / browser_type Exercise the page the way the test will, to confirm a flow works before writing it down.
browser_take_screenshot Visual sanity check when the accessibility tree alone doesn't explain a layout.
browser_console_messages Surface JS errors — feeds directly into the bug-hunter subagent on the Maintenance page.
browser_network_requests Confirm which requests a page actually fires, useful for asserting on API calls in a test.

What You Get Back Result

Because the model actually looked at the live accessibility tree via browser_snapshot instead of guessing, the test it writes back matches real markup — see the sample test on the home page for the shape of what comes out. This grounding is the whole point: an agent that never opens the page will happily invent a #main-nav selector that doesn't exist.
