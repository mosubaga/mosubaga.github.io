What MCP Servers Does the Agent Need?

Instead of one monolithic script, split the agent's capabilities into small MCP servers — each one a standalone tool provider that any host (a custom Python orchestrator, Claude Desktop, or another agent) can call the same way. This keeps textbook search, problem generation, grading, scheduling, and progress tracking independently testable and swappable.

Five MCP Servers for the Study Agent Overview

Server Tools Purpose
textbook-rag search_textbook, get_page Retrieves grounded excerpts from the Chroma index built in the RAG step.
problem-setter generate_problem_set, regenerate_problem Turns retrieved excerpts into a problem set with an answer key.
grader check_solution, request_hint Checks a student's answer and produces graduated hints instead of the solution.
scheduler get_next_topic, mark_day_complete Tracks syllabus pacing so the right topic is chosen each day of the semester.
progress-tracker log_attempt, get_weak_topics Records outcomes per student and surfaces topics that need review.

Building the textbook-rag Server FastMCP

This wraps the Chroma retrieval from the RAG page as two MCP tools. Any client that connects to this server can now call search_textbook without knowing anything about embeddings or vector databases.

textbook_rag_server.py 
# pip install "mcp[cli]" chromadb ollama
from mcp.server.fastmcp import FastMCP
import chromadb

mcp = FastMCP("Textbook RAG Service")
client = chromadb.PersistentClient(path="./vector_store")
collection = client.get_collection("textbooks")

@mcp.tool()
def search_textbook(subject: str, query: str, k: int = 5) -> list[dict]:
    """Search one subject's textbook for excerpts relevant to a topic.

    subject: one of "calculus", "linear_algebra", "probability_statistics"
    query:   the topic or concept to search for
    """
    result = collection.query(query_texts=[query], n_results=k, where={"subject": subject})
    return [
        {"text": doc, "page": meta["page"]}
        for doc, meta in zip(result["documents"][0], result["metadatas"][0])
    ]

@mcp.tool()
def get_page(subject: str, page: int) -> list[str]:
    """Return every indexed chunk for a specific textbook page (for citation lookups)."""
    result = collection.get(where={"$and": [{"subject": subject}, {"page": page}]})
    return result["documents"]

if __name__ == "__main__":
    # stdio transport — a local orchestrator starts this as a subprocess
    mcp.run(transport="stdio")

Wiring All Five Servers Together Config

The orchestrator (the daily job described on the Subagents page) discovers every tool server from one config file, so adding a sixth server later means adding one entry — not touching the orchestrator's code. 

{
  "mcpServers": {
    "textbook-rag": {
      "command": "python",
      "args": ["textbook_rag_server.py"]
    },
    "problem-setter": {
      "command": "python",
      "args": ["problem_setter_server.py"]
    },
    "grader": {
      "command": "python",
      "args": ["grader_server.py"]
    },
    "scheduler": {
      "command": "python",
      "args": ["scheduler_server.py"]
    },
    "progress-tracker": {
      "command": "python",
      "args": ["progress_tracker_server.py"]
    }
  }
}

Safety Checklist for These Servers Important

  • grader should never trust an LLM's judgment alone for numeric answers — verify with sympy first, then use the LLM to explain why.
  • scheduler and progress-tracker write to a student database — scope their file/DB access to that data directory only.
  • Log every generate_problem_set call so a teacher can review what the agent produced.
  • Rate-limit the daily job (one generation per subject per day) so a bug can't spin up unlimited problem sets.
  • Keep read-only tools (search_textbook, get_weak_topics) separate from tools that write state.
