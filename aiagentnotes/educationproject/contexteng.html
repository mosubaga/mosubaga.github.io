Context Engineering for a Semester-Long Agent

Running for ~100 days is the hard part of this project. If the agent naively appends every day's transcript to the next day's prompt, by week 10 it's feeding a bloated, stale history into a small local model — and quality degrades. Context Engineering is what keeps day 97 as sharp as day 1.

The Problem: 100 Days of Accumulated State Why It Matters

A local model has a smaller, more crowded context window than a hosted frontier model, and it suffers from context rot — comprehension quietly degrading as input grows — sooner than a larger model would. A semester-long agent that just keeps appending history will eventually spend most of its context window re-reading January while trying to write April's problem set.

The Three Domains, Applied to This Agent

Acquisition

Each day, pull only what's relevant: today's syllabus topic, the textbook excerpts that match it (see RAG), and this one student's weak-topic list — never the whole book, never the whole semester.

Processing

Subagent isolation (see Subagents) keeps the Tutor's context free of the answer key, and lets the Grader's sympy check result feed back into the LLM before it commits to a correctness judgment — a small self-check loop.

Management

Semester-long state lives outside the LLM entirely — in SQLite — and gets compressed into a short weekly digest before it's ever shown to a model again.

Three Strategies That Do the Work

Daily Context Reset

Every day's job starts a brand-new context. It's rebuilt fresh from the database each morning — never a running chat log across ~100 sessions.

Progressive Hint Disclosure

The Tutor's prompt enforces a hint ladder — conceptual nudge, then a partial step, then a full worked solution — released only across repeated wrong attempts, not all at once.

Weekly Digests

Fifteen weeks of raw attempts compress into fifteen 3-sentence digests — a bounded, roughly constant-size summary instead of unbounded growth.

The Daily Context Budget Practical Guide

What actually goes into the Problem-Setter's prompt each morning, in priority order:

Role & rules — the subagent's fixed system prompt.

Today's syllabus topic — one line from the scheduler.

Retrieved excerpts — the top 5 chunks from RAG, capped by token budget.

Student digest — 3-4 sentences, not the full attempt history.

Output format — the JSON schema the response must follow.

Assembling a Trimmed, Cached Daily Context Python

Two habits keep the daily run both correct and fast: trimming retrieved excerpts to a token budget instead of dumping everything, and reusing keep_alive so Ollama keeps the model and the repeated system prompt warm across the whole batch of students that morning.

build_daily_context.py 
import sqlite3
import ollama

MAX_EXCERPT_WORDS = 700  # keep retrieval from crowding out the digest

def load_student_digest(student_id: str) -> str:
    # The digest is pre-summarized weekly — never the raw attempt log.
    conn = sqlite3.connect("students.db")
    row = conn.execute(
        "SELECT digest FROM student_digests WHERE student_id = ? "
        "ORDER BY week DESC LIMIT 1",
        (student_id,),
    ).fetchone()
    return row[0] if row else "No prior activity."

def trim_excerpts(excerpts: list[dict], max_words: int = MAX_EXCERPT_WORDS) -> str:
    kept, total = [], 0
    for e in excerpts:
        words = len(e["text"].split())
        if total + words > max_words:
            break
        kept.append(f"[p.{e['page']}] {e['text']}")
        total += words
    return "\n\n".join(kept)

def build_context(student_id: str, topic: str, excerpts: list[dict]) -> str:
    digest = load_student_digest(student_id)
    grounding = trim_excerpts(excerpts)
    return (
        f"Topic: {topic}\n\n"
        f"Student summary: {digest}\n\n"
        f"Textbook excerpts:\n{grounding}"
    )

def run_for_batch(students: list[dict], system_prompt: str) -> None:
    for s in students:
        context = build_context(s["id"], s["topic"], s["excerpts"])
        ollama.chat(
            model="qwen2.5:7b",
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": context},
            ],
            # Keep the model (and its processed system prompt) warm for the
            # whole class run instead of reloading it per student.
            keep_alive="30m",
        )

Closing the Loop: Cron for a Semester Automation

Goal items 4 and 5 — daily, automatic, for a whole semester — are satisfied by a plain cron entry. Context engineering is what makes running it 100 times in a row not degrade in quality. 

# crontab -e — run every weekday morning at 6am, semester-long
0 6 * * 1-5 cd /opt/study-agent && python daily_job.py >> logs/daily.log 2>&1
