Mastering Go Concurrency

Concurrency is the ability to run and make progress on multiple smaller components of a larger program simultaneously. Go implements concurrency using goroutines and channels based on the Communicating Sequential Processes (CSP) model.

1. Goroutines: Lightweight Threads

A goroutine is the smallest executable entity in Go. They are managed by the Go runtime and are extremely resource-efficient, requiring only a few kilobytes of stack space that can grow or shrink as needed.

Launching a Goroutine

You create a goroutine by prefixing a function call with the go keyword. This causes the function to run asynchronously in the background.

package main
import (
    "fmt"
    "time"
)

func sayHello() {
    fmt.Println("Hello from a goroutine!")
}

func main() {
    go sayHello() // Starts a new goroutine
    fmt.Println("Main function continues...")
    time.Sleep(time.Millisecond * 10) // Wait briefly so the goroutine can finish
}

2. Synchronizing with WaitGroups

Because the main function does not wait for background goroutines to finish before exiting, we use a sync.WaitGroup to coordinate their completion.

  • Add(n): Sets the number of goroutines to wait for.
  • Done(): Decrements the counter; typically called via defer inside the goroutine.
  • Wait(): Blocks until the counter reaches zero.
import (
    "fmt"
    "sync"
)

var wg sync.WaitGroup

func worker(id int) {
    defer wg.Done() // Signal completion
    fmt.Printf("Worker %d is working...\n", id)
}

func main() {
    for i := 1; i <= 3; i++ {
        wg.Add(1)
        go worker(i)
    }
    wg.Wait() // Block until all 3 workers call Done()
    fmt.Println("All work complete.")
}

3. Channels: Communication Pipes

Channels act as the "glue" that allows goroutines to exchange data safely without explicit locking.

  • Unbuffered Channels: Synchronous; the sender blocks until a receiver is ready.
  • Buffered Channels: Asynchronous; they can hold a limited number of values before blocking the sender.
func main() {
    // Unbuffered channel
    ch := make(chan string)

    go func() {
        ch <- "Data from goroutine" // Send
    }()

    msg := <-ch // Receive (blocks until data is sent)
    fmt.Println(msg)
}

4. Advanced Concurrency Design Patterns

Pattern A: Worker Pools

A worker pool processes many tasks concurrently while controlling the level of parallelism. It uses a shared channel as a work queue for a fixed number of worker goroutines.

func workerPool(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs {
        fmt.Printf("Worker %d processing job %d\n", id, j)
        results <- j * 2
    }
}

func main() {
    jobs := make(chan int, 100)
    results := make(chan int, 100)

    // Start 3 workers
    for w := 1; w <= 3; w++ {
        go workerPool(w, jobs, results)
    }

    // Send 5 jobs
    for j := 1; j <= 5; j++ {
        jobs <- j
    }
    close(jobs) // Signal no more jobs are coming

    // Collect exactly one result for each submitted job
    for a := 1; a <= 5; a++ {
        fmt.Println("result:", <-results)
    }
}

Pattern B: Fan-Out / Fan-In

Fan-out launches multiple goroutines to perform work in parallel. Fan-in collects and aggregates their outputs into a single stream.

// Example Fan-In function (requires: import "sync")
func fanIn(sources ...<-chan string) <-chan string {
    var wg sync.WaitGroup
    out := make(chan string)

    output := func(c <-chan string) {
        defer wg.Done()
        for n := range c {
            out <- n
        }
    }

    wg.Add(len(sources))
    for _, c := range sources {
        go output(c)
    }

    // Close output once all inputs are finished
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

5. Combining Waits and Channels for Efficiency

The most efficient processes often combine WaitGroup for tracking lifecycle and channels for result collection. Using them together prevents race conditions and ensures "clean" termination where no result is lost.

The "Result Aggregator" Pattern

In this pattern, you use a WaitGroup to track workers and a separate goroutine to Wait() and then close() the result channel. This allows the main goroutine to use a range loop over the results channel without deadlocking.

func main() {
    results := make(chan int)
    var wg sync.WaitGroup

    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(val int) {
            defer wg.Done()
            results <- val * val // Send result to channel
        }(i)
    }

    // CRITICAL: Combining Wait and Close in a side-goroutine
    go func() {
        wg.Wait()      // Wait for all workers to finish
        close(results) // Close channel so range loop below can terminate
    }()

    // Efficiently aggregate results as they arrive
    for res := range results {
        fmt.Println("Received:", res)
    }
}

The Select Statement

The select statement lets a goroutine wait on multiple channel operations, proceeding with whichever is ready first. This is essential for implementing timeouts and non-blocking communication.

select {
case msg := <-ch1:
    fmt.Println("Received from ch1:", msg)
case <-time.After(time.Second): // Timeout pattern
    fmt.Println("Timed out!")
}

Summary: Goroutines provide the execution power, WaitGroups manage the lifecycle, and Channels handle the data flow. Combining them correctly is the key to building high-performance, deadlock-free Go applications.

6. The Go Runtime and Scheduler

Understanding how Go manages concurrency under the hood is critical for high-performance applications.

  • M:N Scheduling: Go uses an M:N scheduler where m goroutines are multiplexed onto n OS threads, making goroutines far cheaper than OS threads.
  • The GOMAXPROCS Variable: Controls the number of OS threads that can execute user-level Go code simultaneously. The runtime chooses a sensible default, and explicit tuning is rarely needed.
  • Work-Stealing Algorithm: Underutilized logical processors "steal" tasks from others to automatically balance the load across cores.
import (
    "fmt"
    "runtime"
)

func main() {
    // Query or set the number of logical CPUs in use
    fmt.Println("CPUs:", runtime.NumCPU())
    runtime.GOMAXPROCS(runtime.NumCPU()) // Default; explicit setting is rarely needed
}

7. Advanced Synchronization Primitives

Beyond Mutex and WaitGroup, the sync package provides specialized tools:

  • Condition Variables (sync.Cond): A goroutine waits on a condition via Wait() and is woken by Signal() (one goroutine) or Broadcast() (all waiters).
  • Once (sync.Once): Ensures a piece of code, such as configuration loading or a singleton, runs exactly once regardless of how many goroutines call it.
  • Atomic Operations (sync/atomic): Lock-free primitives for thread-safe counters and flags; faster than a mutex for simple operations.
import "sync"

type Config struct {
    DSN string
}

var once sync.Once
var instance *Config

func GetConfig() *Config {
    once.Do(func() {
        instance = &Config{DSN: "postgres://localhost/app"} // Runs exactly once
    })
    return instance
}

8. Robust Error and Lifecycle Management

Professional concurrent code requires lifecycle control beyond just launching goroutines.

  • Error Groups (errgroup): Tracks a collection of goroutines and returns the first non-nil error encountered, while cancelling the shared context automatically.
  • The Context Package: Essential for propagating deadlines, cancellation signals, and request-scoped values across API boundaries.
  • UNIX Signal Handling: Use channels to catch SIGINT / SIGTERM for graceful shutdowns.
import (
    "context"
    "fmt"
    "golang.org/x/sync/errgroup"
)

func fetchData(ctx context.Context, url string) error {
    fmt.Println("Fetching", url)
    return nil // Replace with real HTTP call
}

func main() {
    g, ctx := errgroup.WithContext(context.Background())

    g.Go(func() error { return fetchData(ctx, "https://api.example.com/a") })
    g.Go(func() error { return fetchData(ctx, "https://api.example.com/b") })

    if err := g.Wait(); err != nil {
        fmt.Println("Error:", err)
    }
}

9. Safety and Performance Optimization

  • Race Detector: The -race flag instruments the binary to detect concurrent unsynchronized memory accesses at runtime.
  • Deadlock Analysis: Avoid circular waits, such as goroutine A waiting for B while B waits for A. Consistent lock ordering and timeouts prevent this.
  • Throttling via Semaphores: Use a buffered channel as a semaphore to cap the number of goroutines accessing a resource concurrently.
  • Testing with t.Parallel(): Run unit tests concurrently to surface synchronization bugs early.
// Run with the race detector:
// $ go test -race ./...
// $ go run -race main.go

type Task struct {
    ID int
}

func process(t Task) {
    fmt.Println("processing", t.ID)
}

// Buffered channel as a semaphore (max 5 concurrent workers)
sem := make(chan struct{}, 5)

for _, task := range tasks {
    sem <- struct{}{}        // Acquire
    go func(t Task) {
        defer func() { <-sem }() // Release
        process(t)
    }(task)
}

10. Specialized Design Patterns

The Pipeline Pattern

Connect goroutines so the output of one stage feeds directly into the input of the next via channels. Each stage is independently concurrent.

func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out)
    }()
    return out
}

func main() {
    for n := range square(generate(2, 3, 4)) {
        fmt.Println(n) // 4, 9, 16
    }
}

The Quit Channel Pattern

Use a dedicated channel to signal one or more goroutines to stop cleanly instead of killing them externally.

func worker(jobs <-chan int, quit <-chan struct{}) {
    for {
        select {
        case j, ok := <-jobs:
            if !ok {
                return
            }
            fmt.Println("Processing", j)
        case <-quit:
            fmt.Println("Worker stopping")
            return
        }
    }
}

func main() {
    jobs := make(chan int, 10)
    quit := make(chan struct{})

    go worker(jobs, quit)

    jobs <- 1
    jobs <- 2
    close(quit) // Signal the worker to stop
}

Monitor Goroutine

A safer alternative to shared memory: a dedicated goroutine owns the data, and others communicate with it via channels to read or update state, eliminating the need for mutexes.

type request struct {
    key   string
    reply chan string
}

func monitor(store map[string]string, reqs <-chan request) {
    for req := range reqs {
        req.reply <- store[req.key]
    }
}

func main() {
    store := map[string]string{"name": "Gopher"}
    reqs := make(chan request)
    go monitor(store, reqs)

    reply := make(chan string)
    reqs <- request{key: "name", reply: reply}
    fmt.Println(<-reply) // Gopher
}
