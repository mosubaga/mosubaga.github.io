Go Packages and Modules

In Go, code is organized into packages, and these packages are grouped into modules. This guide provides a detailed look at how to manage dependencies and project structure using the Go module system.

1. Understanding Packages

A package is a collection of source files located in the same directory that share the same package declaration. They define boundaries between different concerns and control which functions and types are visible to other parts of the program.

Rules for Packages:

  • Main Package: The main package is special and indicates that the compiler should produce an executable program.
  • Visibility (Exporting): Identifiers starting with an uppercase letter are exported (public); those starting with a lowercase letter are unexported (private).
  • Naming: By convention, package names match their directory name and should be short, lowercase, single words.
mathutils/adder.go
package mathutils // Package declaration matches directory name

// Add is exported because it starts with an uppercase letter
func Add(a, b int) int {
    return a + b
}

// subtract is private to this package
func subtract(a, b int) int {
    return a - b
}

2. Introduction to Modules

A module is a collection of related Go packages versioned together as a single unit. It is defined by a go.mod file at its root, which tracks the module's path and its dependency requirements. Modules allow you to build projects outside of the traditional GOPATH.

Initializing a Module:

To start a new project, navigate to your project directory and run:

$ go mod init github.com/username/myproject

The argument github.com/username/myproject is the module path, which serves as a unique identifier and the prefix for all internal packages.

3. The go.mod File in Detail

The go.mod file is both human-readable and machine-editable. It manages all the information the Go toolchain needs to build your project.

Directives in go.mod:

  • module: Declares the module path.
  • go: Specifies the minimum Go version required for the module.
  • require: Lists specific versions of dependencies.
  • replace: Overrides a dependency path, often used for local development.
  • exclude: Prevents specific versions of a module from being used.
go.mod
module github.com/username/myproject // The module path

go 1.26 // Minimum Go language version

require (
    github.com/google/uuid v1.6.0 // Direct dependency
    golang.org/x/text v0.14.0 // Direct dependency
    github.com/mattn/go-sqlite3 v1.14.17 // indirect
)

// The replace directive points a dependency to a local folder
replace github.com/username/othermodule => ../local-version
Security Tip: Alongside go.mod, Go maintains a go.sum file. It contains cryptographic checksums to ensure that your dependencies haven't been tampered with and that every developer gets the exact same code.

4. Common Module Commands

The go command includes several subcommands for module management:

  • go get <path>: Adds or changes a library dependency in go.mod. Use go install package@version for command-line tools.
  • go mod tidy: Removes unused dependencies and adds missing ones to keep go.mod clean.
  • go mod edit: Allows you to modify the go.mod file from the command line (e.g., adding a replace directive).
  • go list -m all: Lists the current module and all its dependencies.

5. Practical Code Example: Using Internal Packages

When you have code that should not be visible to external users of your module, place it in an internal/ directory. The Go compiler forbids other modules from importing packages within an internal directory.

Project Structure:

myproject/
|-- go.mod
|-- main.go
|-- pkg/
|   `-- publicapi/
|       `-- api.go       // Reusable by others
`-- internal/
    `-- secrets/
        `-- database.go  // Only usable by myproject

Importing within the same module:

To import your own packages, use the full module path defined in go.mod.

main.go
package main

import (
    "fmt"
    "github.com/username/myproject/pkg/publicapi" // Full path required
)

func main() {
    publicapi.Start() // Calling an exported function
}
Pro Tip: Use the blank identifier (_) in an import to only trigger a package's init() function without using its exported symbols. This is common for database drivers.
