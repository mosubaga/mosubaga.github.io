SQL Query Optimization

Query optimization is the process of improving SQL query performance to reduce execution time, minimize resource usage, and return results faster. Even small optimizations can make a huge difference when dealing with large datasets.

This guide covers both query-level optimizations and database-level optimizations to help you build faster, more efficient applications.

1. Use Indexes Wisely

Indexes are the most powerful tool for speeding up queries. They work like a book's index, allowing the database to find data without scanning every row.

Creating Indexes

-- Create an index on frequently queried columns
CREATE INDEX idx_email ON users(email);

-- Create a composite index for multiple columns
CREATE INDEX idx_name_date ON employees(last_name, hire_date);

-- Create a unique index
CREATE UNIQUE INDEX idx_username ON users(username);

‚ùå Without Index

-- Full table scan - SLOW
SELECT * FROM users 
WHERE email = 'john@example.com';
-- Scans all rows

‚úÖ With Index

-- Index lookup - FAST
-- After creating index on email
SELECT * FROM users 
WHERE email = 'john@example.com';
-- Direct lookup

üí° Index Best Practices

  • Index columns used in WHERE, JOIN, ORDER BY, and GROUP BY clauses
  • Don't over-index - each index slows down INSERT/UPDATE/DELETE operations
  • Use composite indexes when queries filter on multiple columns together
  • Put the most selective column first in composite indexes
  • Avoid indexing columns with low cardinality (few unique values)

2. Select Only What You Need

Avoid using SELECT * - it retrieves all columns even if you don't need them, wasting network bandwidth and memory.

‚ùå Bad Practice

-- Retrieves all columns unnecessarily
SELECT * FROM employees 
WHERE department = 'Sales';

‚úÖ Good Practice

-- Only retrieve needed columns
SELECT id, name, email 
FROM employees 
WHERE department = 'Sales';

3. Optimize WHERE Clauses

How you write WHERE clauses can significantly impact query performance.

Avoid Functions on Indexed Columns

‚ùå Prevents Index Usage

-- Function on column prevents index use
SELECT * FROM employees 
WHERE YEAR(hire_date) = 2023;

SELECT * FROM users 
WHERE UPPER(email) = 'JOHN@EXAMPLE.COM';

‚úÖ Index-Friendly

-- Rewrite to use index
SELECT * FROM employees 
WHERE hire_date >= '2023-01-01' 
  AND hire_date < '2024-01-01';

SELECT * FROM users 
WHERE email = 'john@example.com';
-- Store emails in lowercase

Use EXISTS Instead of IN with Subqueries

‚ùå Slower

-- IN with subquery
SELECT * FROM customers 
WHERE id IN (
    SELECT customer_id 
    FROM orders 
    WHERE order_date > '2024-01-01'
);

‚úÖ Faster

-- EXISTS stops at first match
SELECT * FROM customers c
WHERE EXISTS (
    SELECT 1 
    FROM orders o 
    WHERE o.customer_id = c.id 
      AND o.order_date > '2024-01-01'
);

Use UNION ALL Instead of UNION When Possible

‚ùå Slower (removes duplicates)

-- UNION removes duplicates - requires sorting
SELECT name FROM customers
UNION
SELECT name FROM suppliers;

‚úÖ Faster (keeps duplicates)

-- UNION ALL doesn't sort/dedupe
SELECT name FROM customers
UNION ALL
SELECT name FROM suppliers;

4. Optimize JOINs

JOINs can be expensive operations. Here's how to make them faster.

Always Index Foreign Keys

-- Index the foreign key column
CREATE INDEX idx_customer_id ON orders(customer_id);

-- Now this JOIN will be much faster
SELECT c.name, o.order_date, o.total
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE o.order_date > '2024-01-01';

Join on Indexed Columns

‚ùå Non-Indexed Column

-- Joining on non-indexed column
SELECT * FROM users u
JOIN profiles p ON LOWER(u.email) = LOWER(p.email);

‚úÖ Indexed Column

-- Join on primary/foreign key
SELECT * FROM users u
JOIN profiles p ON u.id = p.user_id;

Filter Before Joining

‚ùå Filter After Join

-- Joins all rows then filters
SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.order_date > '2024-01-01';

‚úÖ Filter Before Join

-- Filter first to reduce rows joined
SELECT * FROM (
    SELECT * FROM orders 
    WHERE order_date > '2024-01-01'
) o
JOIN customers c ON o.customer_id = c.id;

5. Use LIMIT to Reduce Results

When you don't need all results, use LIMIT to reduce the amount of data processed and transferred.

Pagination with LIMIT and OFFSET

-- Get first 20 results
SELECT * FROM products 
ORDER BY created_at DESC 
LIMIT 20;

-- Get next 20 results (page 2)
SELECT * FROM products 
ORDER BY created_at DESC 
LIMIT 20 OFFSET 20;

‚ö° Performance Tip

For better pagination performance on large datasets, use keyset pagination instead of OFFSET:

-- Better for large offsets
SELECT * FROM products 
WHERE id > 1000  -- Last ID from previous page
ORDER BY id 
LIMIT 20;

6. Avoid N+1 Query Problems

The N+1 problem occurs when you make one query to get a list, then make N additional queries for each item in that list.

‚ùå N+1 Queries (Slow)

-- Query 1: Get all customers
SELECT * FROM customers;

-- Then for EACH customer (N queries):
SELECT * FROM orders WHERE customer_id = 1;
SELECT * FROM orders WHERE customer_id = 2;
SELECT * FROM orders WHERE customer_id = 3;
-- ... potentially hundreds of queries!

‚úÖ Single Query with JOIN (Fast)

-- One query gets everything
SELECT c.*, o.*
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;

-- Or use IN clause
SELECT * FROM orders 
WHERE customer_id IN (1, 2, 3, ...);

7. Database-Level Optimizations

Beyond query optimization, you can optimize the database itself.

Analyze and Update Statistics

-- MySQL: Analyze table to update statistics
ANALYZE TABLE employees;

-- PostgreSQL: Analyze table
ANALYZE employees;

-- SQL Server: Update statistics
UPDATE STATISTICS employees;

Database optimizers use statistics to choose the best execution plan. Keep statistics up to date, especially after large data changes.

Partitioning Large Tables

-- Partition by date range
CREATE TABLE orders (
    id INT,
    customer_id INT,
    order_date DATE,
    total DECIMAL(10, 2)
)
PARTITION BY RANGE (YEAR(order_date)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025)
);

Partitioning divides large tables into smaller, more manageable pieces, improving query performance when queries target specific partitions.

Use Appropriate Data Types

‚ùå Inefficient

-- Using VARCHAR for numbers
CREATE TABLE products (
    id VARCHAR(50),
    price VARCHAR(20),
    in_stock VARCHAR(10)
);

‚úÖ Efficient

-- Use appropriate data types
CREATE TABLE products (
    id INT,
    price DECIMAL(10, 2),
    in_stock BOOLEAN
);

Normalize vs. Denormalize

Normalization: Reduces data redundancy, saves storage, maintains data integrity. Good for write-heavy applications.

-- Normalized: separate tables
CREATE TABLE customers (id INT, name VARCHAR(100));
CREATE TABLE orders (id INT, customer_id INT, total DECIMAL(10,2));

-- Requires JOIN to get customer with orders
SELECT c.name, o.total
FROM customers c
JOIN orders o ON c.id = o.customer_id;

Denormalization: Duplicates data to reduce JOINs. Good for read-heavy applications.

-- Denormalized: combined data
CREATE TABLE orders (
    id INT, 
    customer_id INT,
    customer_name VARCHAR(100),  -- Duplicated data
    total DECIMAL(10,2)
);

-- No JOIN needed - faster reads
SELECT customer_name, total FROM orders;

8. Use Query Execution Plans

Execution plans show you exactly how the database executes your query, helping you identify bottlenecks.

Analyzing Query Plans

-- MySQL
EXPLAIN SELECT * FROM employees WHERE department = 'Sales';

-- PostgreSQL
EXPLAIN ANALYZE SELECT * FROM employees WHERE department = 'Sales';

-- SQL Server
SET SHOWPLAN_TEXT ON;
GO
SELECT * FROM employees WHERE department = 'Sales';
GO

üîç What to Look For

  • Table Scans: Bad - means no index is used
  • Index Scans: Good - index is being used
  • Rows Examined: Lower is better
  • Execution Time: Your optimization target
  • Nested Loops: Can be slow with large datasets

9. Caching Strategies

Sometimes the best optimization is not running the query at all.

Query Result Caching

-- MySQL Query Cache (older versions)
SELECT SQL_CACHE * FROM products WHERE category = 'Electronics';

-- Application-level caching (pseudocode)
-- Check cache first
result = cache.get('products:electronics');
if (result == null) {
    result = db.query('SELECT * FROM products WHERE category = ?', 'Electronics');
    cache.set('products:electronics', result, 3600); // Cache for 1 hour
}
return result;

Materialized Views

-- PostgreSQL: Create a materialized view
CREATE MATERIALIZED VIEW sales_summary AS
SELECT 
    product_id,
    COUNT(*) as total_orders,
    SUM(quantity) as total_quantity,
    SUM(total) as total_revenue
FROM orders
GROUP BY product_id;

-- Refresh periodically
REFRESH MATERIALIZED VIEW sales_summary;

-- Query the materialized view (very fast)
SELECT * FROM sales_summary WHERE product_id = 123;

10. Batch Operations

When performing multiple operations, batch them together to reduce overhead.

‚ùå Individual Inserts

-- Multiple round trips to database
INSERT INTO products (name, price) VALUES ('Product 1', 10.99);
INSERT INTO products (name, price) VALUES ('Product 2', 15.99);
INSERT INTO products (name, price) VALUES ('Product 3', 20.99);
-- ... 1000 more inserts

‚úÖ Batch Insert

-- Single query with multiple values
INSERT INTO products (name, price) VALUES 
    ('Product 1', 10.99),
    ('Product 2', 15.99),
    ('Product 3', 20.99),
    ('Product 4', 25.99);
-- Much faster!
