Closures in JavaScript

A closure is one of the most important JavaScript concepts for building real web applications. It explains how functions remember variables from the scope where they were created.

What is a Closure?

A closure is created when a function remembers and continues to access variables from its outer scope, even after the outer function has finished running.

function counter() {
  let count = 0;

  return function () {
    count++;
    return count;
  };
}

const increment = counter();

console.log(increment()); // 1
console.log(increment()); // 2
console.log(increment()); // 3

The inner function still has access to count, even though counter() already finished executing. That remembered access is the closure.

The Key Idea

JavaScript uses lexical scope. That means a function's available variables are determined by where the function is written in the source code, not just where it is called.

function outer() {
  const message = "Hello from outer";

  function inner() {
    console.log(message);
  }

  return inner;
}

const savedFunction = outer();
savedFunction(); // Hello from outer
Important: The function inner carries its surrounding environment with it. It does not copy the value once; it keeps access to the variable itself.

Why Closures Matter

Private State

Closures let you hide data so outside code cannot directly modify it.

Callbacks

Event handlers and timers often use closures to remember values.

Factories

Functions can create customized functions with remembered configuration.

Modules

Closures help separate public APIs from private implementation details.

Closures for Private Variables

Before modern JavaScript classes and private fields, closures were one of the most common ways to create private data.

function createBankAccount(initialBalance) {
  let balance = initialBalance;

  return {
    deposit(amount) {
      balance += amount;
      return balance;
    },

    withdraw(amount) {
      if (amount > balance) {
        return "Insufficient funds";
      }
      balance -= amount;
      return balance;
    },

    getBalance() {
      return balance;
    }
  };
}

const account = createBankAccount(100);

console.log(account.deposit(50));   // 150
console.log(account.withdraw(25));  // 125
console.log(account.getBalance());  // 125
console.log(account.balance);       // undefined

The variable balance is protected. The only way to change it is through the returned methods.

Closures in Web App Development

Closures are not just a theory topic. They appear constantly in real web apps.

1. Event Handlers

function setupButton(username) {
  const button = document.querySelector("#saveButton");

  button.addEventListener("click", function () {
    console.log(`Saving data for ${username}`);
  });
}

setupButton("John");

The click handler remembers username because it forms a closure.

2. Search Boxes and Debouncing

function debounce(fn, delay) {
  let timerId;

  return function (...args) {
    clearTimeout(timerId);

    timerId = setTimeout(() => {
      fn(...args);
    }, delay);
  };
}

const search = debounce(function (text) {
  console.log("Searching for:", text);
}, 500);

search("j");
search("ja");
search("jav");

The returned function remembers timerId. This is how debounce avoids running the search function too often.

3. Caching API Results

function createUserService() {
  const cache = new Map();

  return async function getUser(id) {
    if (cache.has(id)) {
      return cache.get(id);
    }

    const response = await fetch(`/api/users/${id}`);
    const user = await response.json();

    cache.set(id, user);
    return user;
  };
}

const getUser = createUserService();

The cache variable stays private but remains available to getUser.

Closures in React

React developers use closures constantly, even when they do not explicitly think about them.

function Counter() {
  const [count, setCount] = React.useState(0);

  function handleClick() {
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

The function handleClick closes over count and setCount. This is why event handlers inside components can access component state.

In React, closures are powerful but can also cause stale state bugs if an old function remembers an old value. This is common with timers, effects, and asynchronous callbacks.

A Common Closure Mistake

Older JavaScript code using var often produces confusing closure behavior in loops.

for (var i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i);
  }, 100);
}

// 3
// 3
// 3

Because var is function-scoped, all three callbacks share the same i.

Using let fixes this because let creates a new binding for each loop iteration.

for (let i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i);
  }, 100);
}

// 0
// 1
// 2

Closures and Memory

Closures keep referenced variables alive. This is useful, but it can also cause memory problems if you accidentally keep large objects around.

function createProcessor() {
  const largeData = new Array(1_000_000).fill("data");

  return function process(index) {
    return largeData[index];
  };
}

const processor = createProcessor();

As long as processor exists, largeData cannot be garbage collected.

Best Practices

  • Use closures to protect private state.
  • Use closures for event handlers, callbacks, debounce, throttle, and caching.
  • Avoid storing huge data structures inside long-lived closures unless necessary.
  • Prefer let and const over var to avoid loop-related closure bugs.
  • In React, watch for stale closures in useEffect, timers, and async callbacks.
