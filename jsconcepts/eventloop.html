The Event Loop in JavaScript

The event loop explains how JavaScript handles asynchronous work such as timers, user events, network requests, and promises without blocking the browser's main thread.

What is the Event Loop?

The event loop is the mechanism that lets JavaScript run synchronous code first, then come back later to handle asynchronous callbacks.

JavaScript usually runs on one main thread in the browser. That means only one piece of JavaScript code executes at a time. The event loop coordinates what should run next.

Main idea: JavaScript does not stop and wait for slow work like timers or network requests. The browser handles that work in the background, then places callbacks in a queue. The event loop sends those callbacks back to the call stack when the stack is empty.

The Pieces Involved

Call Stack

Where currently running JavaScript functions are placed. The top function runs first.

Browser APIs

Features provided by the browser, such as setTimeout, DOM events, and fetch.

Task Queue

Stores callbacks from timers, DOM events, and other tasks waiting to run.

Microtask Queue

Stores promise callbacks, such as .then(), .catch(), and code after await.

Event Loop

Checks whether the call stack is empty. If it is, it moves queued work onto the stack.

Simple Example

console.log("Start");

setTimeout(() => {
  console.log("Timer finished");
}, 0);

console.log("End");

Output:

Start
End
Timer finished

Even though the timer delay is 0, the callback does not run immediately. It must wait until the current synchronous code finishes and the call stack becomes empty.

Step-by-Step Execution

console.log("Start");

setTimeout(() => {
  console.log("Async");
}, 0);

console.log("End");

1. Run synchronous code

console.log("Start"); // prints Start

2. Register the timer

setTimeout(..., 0);

The callback is handed to the browser timer API. JavaScript continues running.

3. Continue synchronous code

console.log("End"); // prints End

4. Event loop checks the queue

After the call stack is empty, the timer callback is moved to the stack.

5. Run the callback

console.log("Async"); // prints Async

Promises and the Microtask Queue

Promises use the microtask queue, which has higher priority than the regular task queue.

console.log("Start");

setTimeout(() => {
  console.log("Timer");
}, 0);

Promise.resolve().then(() => {
  console.log("Promise");
});

console.log("End");

Output:

Start
End
Promise
Timer

The promise callback runs before the timer callback because microtasks are processed before regular tasks.

How async and await Fit In

async and await are built on promises. When JavaScript reaches await, the function pauses, returns control to the rest of the program, and continues later as a microtask.

async function loadUser() {
  console.log("Before fetch");

  const response = await fetch("/api/user");
  const user = await response.json();

  console.log(user.name);
}

loadUser();
console.log("App continues running");

The browser can remain responsive while the network request is in progress.

Why the Event Loop Matters in Web Applications

The event loop is essential to web app development because the browser must stay responsive while your app handles many kinds of work.

Web App Feature How the Event Loop Applies
Button clicks Click handlers wait in the task queue until the current code finishes.
API requests fetch runs asynchronously. The UI can keep working while data loads.
Search boxes Input events can trigger debounced API calls without freezing the page.
Animations Browser rendering happens between event loop turns. Long JavaScript tasks can make animations choppy.
React, Vue, Angular, Svelte Frameworks schedule state updates, rendering work, and effects around the event loop.

Example: Loading Data Without Freezing the UI

In a real app, you might show a loading message while waiting for data from an API.

const button = document.querySelector("#loadButton");
const output = document.querySelector("#output");

button.addEventListener("click", async () => {
  output.textContent = "Loading...";

  try {
    const response = await fetch("/api/products");
    const products = await response.json();

    output.textContent = `Loaded ${products.length} products`;
  } catch (error) {
    output.textContent = "Could not load products";
  }
});

The page does not lock up while fetch is waiting for the server. The event loop allows other events, rendering, and user interactions to continue.

Common Mistake: Blocking the Event Loop

Heavy synchronous code blocks everything else. While this loop is running, the browser cannot respond to clicks, update the UI, or run timers.

button.addEventListener("click", () => {
  for (let i = 0; i < 5_000_000_000; i++) {
    // Expensive work
  }

  console.log("Finished");
});

This can make the app feel frozen.

Best practice: Break large work into smaller chunks, move heavy work to a Web Worker, or perform expensive operations on the server.

Common Mistake: Expecting setTimeout(..., 0) to Run Immediately

setTimeout(() => {
  console.log("Timer");
}, 0);

console.log("Synchronous code");

Output:

Synchronous code
Timer

A zero-delay timer means "run this after the current call stack is clear," not "run this right now."

Best Practices for App Development

  • Use async / await for network requests and asynchronous workflows.
  • Never perform expensive CPU-heavy work directly in event handlers.
  • Use debounce for search inputs and resize events.
  • Use throttle for scroll, mousemove, and drag events.
  • Prefer clear loading states when waiting for asynchronous data.
  • Handle errors with try...catch around awaited operations.
  • Remember that promise callbacks run before timer callbacks.

Key Takeaway

The event loop is what makes JavaScript useful for web apps. It allows JavaScript to start asynchronous work, keep the UI responsive, and then process callbacks later when the call stack is clear.

Remember: Synchronous code first, then microtasks, then regular tasks.
