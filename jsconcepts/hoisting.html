Hoisting in JavaScript

Hoisting explains why some variables and functions can be referenced before their declarations appear in the source code. It is really about how JavaScript prepares memory before running your code.

What is Hoisting?

Hoisting is JavaScript's behavior of processing declarations before executing the code.

Before the first line of your program runs, the JavaScript engine scans the current scope and creates memory for variables and functions.

1. Creation Phase

  • Create the execution context
  • Allocate memory
  • Register variable declarations
  • Register function declarations

2. Execution Phase

  • Execute code line by line
  • Assign values
  • Call functions
  • Update memory
JavaScript does not literally move your code to the top. The engine prepares declarations before execution begins.

Example 1: var

console.log(user);

var user = "John";

Output:

undefined

JavaScript behaves as if the declaration was known first, while the assignment stays in place.

var user;

console.log(user);

user = "John";

The declaration is hoisted. The assignment is not.

Example 2: Function Declarations

greet();

function greet() {
  console.log("Hello");
}

Output:

Hello

Function declarations are fully available during the creation phase, so they can be called before they appear in the file.

Example 3: Function Expressions

greet();

const greet = function () {
  console.log("Hello");
};

This causes a ReferenceError because greet is declared with const. The variable exists, but it cannot be accessed before initialization.

var, let, and const

Keyword Hoisted? Initial Value Before Declaration? Behavior
var Yes undefined Can be read before assignment, but the value is undefined.
let Yes Uninitialized Cannot be accessed before declaration because of the Temporal Dead Zone.
const Yes Uninitialized Same as let, but reassignment is also not allowed.

let and the Temporal Dead Zone

console.log(a);

let a = 10;

Output:

ReferenceError: Cannot access 'a' before initialization

let and const are hoisted, but they remain unavailable until execution reaches the declaration line.

Visualizing Hoisting

console.log(user);

var user = "John";

greet();

function greet() {
  console.log("Hi");
}

Creation Phase

Memory

user  → undefined
greet → function greet() { ... }

Execution Phase

greet();           // Hi

Why JavaScript Has Hoisting

Hoisting allows functions to be called before their definitions appear in the source file. This can make application startup code easier to read.

initializeApp();

function initializeApp() {
  loadConfiguration();
  attachEvents();
  renderUI();
}

How Hoisting Applies to Web App Development

1. Application Startup

initialize();

function initialize() {
  loadConfig();
  attachEvents();
  startApplication();
}

This works because initialize is a function declaration.

2. Event Handlers

button.addEventListener("click", saveUser);

function saveUser() {
  console.log("Saved");
}

The event handler can be registered before its function declaration appears.

3. Modules

// app.js
import { login } from "./auth.js";

login();
// auth.js
export function login() {
  console.log("Logging in...");
}

Modern apps use ES modules. Inside each module, declarations still follow JavaScript's hoisting behavior within that module scope.

4. React Components

function App() {
  function login() {
    console.log("Login");
  }

  return (
    <button onClick={login}>
      Login
    </button>
  );
}

This works because login is a function declaration inside the component.

function App() {
  return (
    <button onClick={login}>
      Login
    </button>
  );

  const login = () => {
    console.log("Login");
  };
}

This version fails because login is a const function expression and cannot be used before initialization.

Common Mistakes

Mistake 1: Assuming undefined means not declared

console.log(a);

var a = 5;

The variable is declared, but it has not been assigned yet.

Mistake 2: Thinking let is not hoisted

let and const are hoisted, but they cannot be accessed before initialization.

Mistake 3: Mixing var and let

var count = 5;

if (true) {
  let count = 10;
}

Different scoping rules can create subtle bugs.

Best Practices

  • Prefer const by default.
  • Use let only when reassignment is needed.
  • Avoid var in new code.
  • Declare variables near their first use.
  • Use function declarations when you intentionally want callable functions throughout the scope.
  • Use function expressions when you want initialization order to be explicit.

Interview Questions

Question 1

console.log(x);

var x = 10;

Answer: It prints undefined because var is hoisted and initialized to undefined.

Question 2

console.log(x);

let x = 10;

Answer: It throws a ReferenceError because x is in the Temporal Dead Zone.

Question 3

function foo() {}

const bar = function () {};

Answer: foo can be called before its declaration. bar cannot be called before initialization.
