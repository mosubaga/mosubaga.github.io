JavaScript Modules

Modules let you split a JavaScript application into smaller files with clear responsibilities. They make code easier to organize, reuse, test, debug, and maintain.

What Is a JavaScript Module?

A module is a JavaScript file that contains its own code and can share selected values with other files. Instead of putting every variable and function into one large script file, modules allow you to divide your application into meaningful parts.

In modern JavaScript, modules usually use export and import. 

// math.js
export function add(a, b) {
  return a + b;
}

// app.js
import { add } from "./math.js";

console.log(add(2, 3)); // 5
The main idea: one file exports something, another file imports it.

Why Modules Matter

Without modules, JavaScript applications can quickly become difficult to manage. Imagine placing all of your login logic, API calls, form validation, UI rendering, shopping cart code, and utility functions into one large file. That file would become hard to understand and risky to modify.

Organization

Each file has a clear purpose, such as auth.js, api.js, or cart.js.

Reuse

Common functions can be imported wherever they are needed.

Encapsulation

Code inside a module stays private unless you explicitly export it.

Maintainability

Smaller files are easier to debug, test, and change.

The Problem Before Modules

Older JavaScript applications often used multiple script tags. This worked, but all scripts shared the same global scope. That could easily cause naming conflicts. 

<script src="user.js"></script>
<script src="cart.js"></script>
<script src="app.js"></script>

If both user.js and cart.js created a global variable named data, one could accidentally overwrite the other. 

// user.js
var data = { name: "Alice" };

// cart.js
var data = [];

// app.js
console.log(data); // Which data is this?

Modules help prevent this because module variables are not automatically global.

Named Exports

A named export exports one or more values by name. The importing file must use the same exported names. 

// validation.js
export function isEmail(value) {
  return value.includes("@");
}

export function isRequired(value) {
  return value.trim().length > 0;
}
// app.js
import { isEmail, isRequired } from "./validation.js";

console.log(isRequired("hello")); // true
console.log(isEmail("test@example.com")); // true

Named exports are useful when a module provides several related functions.

Default Exports

A default export is used when a module mainly exports one primary thing. 

// logger.js
export default function log(message) {
  console.log(`[APP]: ${message}`);
}
// app.js
import log from "./logger.js";

log("Application started");

When importing a default export, you can choose the name used in the importing file. 

import myLogger from "./logger.js";

myLogger("Still works");

Named Export vs Default Export

Feature Named Export Default Export
Syntax export function add() {} export default function add() {}
Import import { add } from "./math.js" import add from "./math.js"
Best for Multiple utilities from one file One main thing from one file
Rename import { add as sum } Name can be chosen freely

Using Modules in the Browser

To use ES Modules directly in the browser, add type="module" to your script tag. 

<script type="module" src="app.js"></script>

Then your JavaScript file can use import and export. 

// app.js
import { showMessage } from "./ui.js";

showMessage("Hello from a module!");
// ui.js
export function showMessage(message) {
  document.body.insertAdjacentHTML("beforeend", `<p>${message}</p>`);
}
Browser modules are automatically deferred. That means they do not block HTML parsing in the same way as a classic script.

Modules and Application Development

In real web applications, modules are used to organize code by responsibility. A small app might look like this: 

src/
├── app.js
├── api/
│   └── usersApi.js
├── components/
│   └── userList.js
├── services/
│   └── authService.js
└── utils/
    └── formatDate.js

Each file has a specific job. 

// api/usersApi.js
export async function fetchUsers() {
  const response = await fetch("/api/users");

  if (!response.ok) {
    throw new Error("Failed to fetch users");
  }

  return response.json();
}
// components/userList.js
export function renderUserList(users) {
  return users
    .map(user => `<li>${user.name}</li>`)
    .join("");
}
// app.js
import { fetchUsers } from "./api/usersApi.js";
import { renderUserList } from "./components/userList.js";

async function startApp() {
  const users = await fetchUsers();
  document.querySelector("#users").innerHTML = renderUserList(users);
}

startApp();

This is much cleaner than putting the API call, rendering logic, and startup logic into one large file.

Modules and Private Code

Code inside a module is private by default. Other files can only access what is exported. 

// authService.js
let currentUser = null;

function saveToken(token) {
  localStorage.setItem("token", token);
}

export function login(user, token) {
  currentUser = user;
  saveToken(token);
}

export function getCurrentUser() {
  return currentUser;
}

In this example, currentUser and saveToken are not directly available outside the module. Only login and getCurrentUser are exported.

This is one reason modules are important for app design: they help you expose a clean public API while hiding implementation details.

Modules and State

A module can hold state. This can be useful, but it must be handled carefully. 

// cartStore.js
let items = [];

export function addItem(item) {
  items.push(item);
}

export function getItems() {
  return [...items];
}

export function clearCart() {
  items = [];
}
// app.js
import { addItem, getItems } from "./cartStore.js";

addItem({ id: 1, name: "Keyboard" });
console.log(getItems());

This pattern is simple and useful for small apps. In larger apps, state management usually needs more structure.

Dynamic Imports

A normal import is static. It is loaded before the module runs. A dynamic import loads code only when needed. 

button.addEventListener("click", async () => {
  const module = await import("./charts.js");
  module.renderChart();
});

Dynamic imports are useful for performance. For example, a dashboard might only load charting code after the user opens the analytics page.

Modules in React, Vue, and Modern Frameworks

Modern frameworks rely heavily on modules. A React component is usually exported from one file and imported into another. 

// Button.jsx
export function Button({ children }) {
  return <button>{children}</button>;
}
// App.jsx
import { Button } from "./Button.jsx";

export default function App() {
  return <Button>Save</Button>;
}

Even though the syntax looks framework-specific because of JSX, the organizing idea is still JavaScript modules.

Common Mistakes

1. Forgetting the file extension in browser modules

// Browser module usually needs the file extension
import { add } from "./math.js";

2. Mixing named and default imports incorrectly

// math.js
export function add(a, b) {
  return a + b;
}

// Wrong
import add from "./math.js";

// Correct
import { add } from "./math.js";

3. Creating circular dependencies

A circular dependency happens when module A imports from module B, and module B also imports from module A. This can make application startup confusing and may produce partially initialized values. 

// a.js
import { b } from "./b.js";

// b.js
import { a } from "./a.js";

4. Making modules too large

If a module does too many things, it becomes hard to reuse and test. Prefer files with clear responsibilities.

Best Practices

  • Use const and let inside modules instead of global variables.
  • Export only what other files truly need.
  • Keep modules focused on one responsibility.
  • Use named exports for utility modules with multiple functions.
  • Use default exports when a file has one main purpose, such as one component or one service.
  • Avoid circular dependencies when possible.
  • Use dynamic imports for expensive code that is not needed immediately.

Practical Mini Project: Modular Todo App

A small todo application could be organized like this: 

todo-app/
├── index.html
├── app.js
├── todoStore.js
└── todoView.js
// todoStore.js
let todos = [];

export function addTodo(text) {
  todos.push({ id: Date.now(), text, done: false });
}

export function getTodos() {
  return [...todos];
}
// todoView.js
export function renderTodos(todos) {
  return todos
    .map(todo => `<li>${todo.text}</li>`)
    .join("");
}
// app.js
import { addTodo, getTodos } from "./todoStore.js";
import { renderTodos } from "./todoView.js";

const form = document.querySelector("#todo-form");
const input = document.querySelector("#todo-input");
const list = document.querySelector("#todo-list");

function updateUI() {
  list.innerHTML = renderTodos(getTodos());
}

form.addEventListener("submit", event => {
  event.preventDefault();
  addTodo(input.value);
  input.value = "";
  updateUI();
});

Interview Questions

Q1. What is a module?

A module is a JavaScript file with its own scope that can export selected values and import values from other modules.

Q2. Why are modules useful?

They help organize code, avoid global variables, improve reuse, and make applications easier to maintain.

Q3. What is the difference between named and default exports?

Named exports are imported using their exported names inside braces. A default export represents the main value of a module and can be imported with any local name.

Q4. What is a dynamic import?

A dynamic import loads a module at runtime using import(). It returns a Promise and is useful for lazy loading.

Summary

JavaScript modules are one of the most important features in modern web development. They allow applications to be split into smaller, reusable, private, and maintainable files. In real application development, modules are used for components, API clients, services, utility functions, state management, and application startup code.

If hoisting explains how declarations are prepared before execution, modules explain how an entire application can be structured across many files.
