Git Troubleshooting

Step-by-step fixes for the most common Git problems developers run into.

Detached HEAD State

You are in "detached HEAD" state when HEAD points directly to a commit rather than a branch. This happens after git checkout [commit-hash] or git checkout [tag]. Any commits you make will be lost once you switch away unless you create a branch.

Option A — Save your work on a new branch:

  1. 1git checkout -b [new-branch-name] — create a branch at the current commit
  2. 2Continue working normally on the new branch

Option B — Discard and return to a branch:

  1. 1git checkout main (or any branch name) — return to a named branch
Merge Conflicts

A merge conflict occurs when two branches have modified the same part of the same file differently. Git cannot automatically decide which change to keep and marks the conflicting sections in the file.

  1. 1git status — list all files with conflicts (shown as "both modified")
  2. 2Open each conflicted file and look for <<<<<<<, =======, and >>>>>>> markers
  3. 3Edit the file to keep the correct code and remove all conflict markers
  4. 4git add [file] — mark each resolved file as resolved
  5. 5git commit — finalize the merge commit

To abort and start over:

  1. 1git merge --abort — cancel the merge and restore the pre-merge state
Accidentally Committed to main (or wrong branch)

You committed work directly to main (or another protected branch) instead of a feature branch. You need to move those commits to a new branch and undo them on main.

  1. 1git branch [feature-branch] — create a new branch at the current commit (your work is now saved here)
  2. 2git reset --hard HEAD~[n] — move main back n commits (replace n with number of commits to undo)
  3. 3git checkout [feature-branch] — switch to your new branch and continue working
Undo Last Commit (keep your changes)

You committed too early or need to revise what's in the commit — but you don't want to lose the actual file changes.

Keep changes staged (ready to re-commit):

  1. 1git reset --soft HEAD~1

Keep changes unstaged (in working tree):

  1. 1git reset HEAD~1 (same as --mixed)

Discard all changes from the last commit:

  1. 1git reset --hard HEAD~1 — WARNING: changes are permanently lost
Forgot to Add Files to Last Commit

You just committed but forgot to include one or more files. If you haven't pushed yet, you can amend the commit to include them.

  1. 1git add [forgotten-file] — stage the missing file
  2. 2git commit --amend --no-edit — add it to the last commit without changing the message

Note: Do not amend commits that have already been pushed to a shared remote branch.

Push Rejected: Non-Fast-Forward / Remote Has New Commits

Someone else pushed commits to the remote branch after your last pull. Git cannot automatically merge them with a simple push.

Option A — Rebase (clean linear history):

  1. 1git pull --rebase — fetch the new commits and replay yours on top
  2. 2Resolve any conflicts if prompted
  3. 3git push

Option B — Merge (preserves branch history):

  1. 1git pull — fetch and merge remote commits
  2. 2Resolve any conflicts if prompted
  3. 3git push
Accidentally Deleted a Branch

You deleted a branch with git branch -D that still had unmerged commits. Git does not immediately garbage-collect those commits, so you can recover them using the reflog.

  1. 1git reflog — find the commit hash of the last commit that was on the deleted branch
  2. 2Look for an entry like HEAD@{n}: commit: [your commit message]
  3. 3git checkout -b [branch-name] [commit-hash] — recreate the branch at that commit
Wrong Commit Message

You made a typo or used the wrong message in the most recent commit and haven't pushed yet.

  1. 1git commit --amend -m "Correct message here" — rewrite the last commit message

For commits already pushed to a shared branch, use git revert instead of amending.

Accidentally Committed a Large File or Sensitive Data

You committed a file you shouldn't have (e.g., a large binary, an .env file, or API keys). If not yet pushed, you can fix it locally.

If not yet pushed:

  1. 1git rm --cached [file] — stop tracking the file (keeps it on disk)
  2. 2Add the file to .gitignore
  3. 3git commit --amend --no-edit — update the last commit to exclude the file

If already pushed (sensitive data — act fast):

  1. 1Immediately rotate/revoke any exposed credentials
  2. 2Use git filter-repo (recommended) or BFG Repo Cleaner to purge the file from all history
  3. 3Force-push all branches: git push --force-with-lease --all
  4. 4Notify all collaborators to re-clone the repository
Wrong Remote URL / Remote Changed

The remote repository was moved, renamed, or you set the wrong URL when cloning.

  1. 1git remote -v — view the current remote URLs
  2. 2git remote set-url origin [new-url] — update the URL
  3. 3git remote -v — verify the change
Error: "refusing to merge unrelated histories"

This happens when you try to merge two repositories that have no common ancestor — for example, after git init on a project and then trying to pull from a freshly created GitHub repo.

  1. 1git pull origin main --allow-unrelated-histories — force Git to merge despite no shared history
  2. 2Resolve any conflicts that arise
  3. 3git push -u origin main
Authentication Failure / Cannot Push or Pull

GitHub and GitLab no longer accept passwords for HTTPS operations — you need a Personal Access Token (PAT) or an SSH key.

Option A — Use an SSH key (recommended):

  1. 1ssh-keygen -t ed25519 -C "your@email.com" — generate a key pair
  2. 2Add the public key (~/.ssh/id_ed25519.pub) to your GitHub/GitLab account settings
  3. 3git remote set-url origin git@github.com:[user]/[repo].git — switch remote to SSH

Option B — Use a Personal Access Token (HTTPS):

  1. 1Create a PAT in your GitHub/GitLab account settings (with "repo" scope)
  2. 2When prompted for a password, enter the PAT instead of your account password
  3. 3git config --global credential.helper store — cache credentials to avoid repeated prompts
Committed on the Wrong Branch — Need to Move Changes

You started working and committed on main when you meant to work on a feature branch. Your changes need to be moved to the correct branch.

Using cherry-pick (moves commits cleanly):

  1. 1git log --oneline -5 — note the commit hash(es) you want to move
  2. 2git checkout [correct-branch] — switch to the target branch
  3. 3git cherry-pick [commit-hash] — apply the commit here
  4. 4git checkout main — go back to main
  5. 5git reset --hard HEAD~1 — remove the commit from main
Conflicts During Rebase

During an interactive or regular rebase, a commit causes a conflict that Git cannot resolve automatically.

To resolve and continue:

  1. 1git status — see which files have conflicts
  2. 2Edit each conflicted file to resolve the conflict markers
  3. 3git add [file] — mark each file as resolved
  4. 4git rebase --continue — proceed to the next commit in the rebase

To abort and restore original state:

  1. 1git rebase --abort — cancel the entire rebase
Repository is Slow or Growing Too Large

Over time, unreferenced objects, large binaries, or deleted history can bloat a repository and slow it down.

  1. 1git gc — run garbage collection to compress and clean the repo
  2. 2git gc --aggressive --prune=now — more thorough cleanup (takes longer)
  3. 3git count-objects -vH — check the repo size before and after
  4. 4Use .gitignore to avoid committing large build artifacts or binaries going forward
  5. 5Consider Git LFS (git lfs track "*.psd") for large files that must be versioned
