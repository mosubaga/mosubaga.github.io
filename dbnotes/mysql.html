MySQLMySQL on Linux

Install MySQL from your distribution's packages, secure the fresh installation, control the service with systemd, and run your first commands in the mysql client.

1. Installing MySQL on Ubuntu / Debian

What it does: downloads and installs the MySQL server package and registers it as a systemd service. When to use it: on any Debian-family distribution (Ubuntu, Debian, Linux Mint, Pop!_OS). 

sudo apt update
sudo apt install mysql-server

apt update refreshes the package index so you get the current version; apt install mysql-server installs the server, the client tools and the systemd unit. On Ubuntu the service usually starts automatically after installation.

Note — Debian and MariaDB On plain Debian, the mysql-server package may be provided by MariaDB, a community fork of MySQL. For learning purposes the commands on this site work the same way on both — see MariaDB below.

2. Installing MySQL on Fedora / RHEL-based systems

When to use it: on Fedora, RHEL, CentOS Stream, Rocky Linux or AlmaLinux, which use the dnf package manager. 

sudo dnf install mysql-server

Unlike Ubuntu, Fedora-family systems usually do not start the service automatically. Start it and enable it at boot: 

sudo systemctl enable --now mysqld

enable --now does two things at once: starts the service immediately and registers it to start on every boot.

3. Checking the MySQL service status

What it does: shows whether the server process is running, when it started, and the most recent log lines. When to use it: after installing, and any time a connection fails — always check the service first. 

sudo systemctl status mysql

or:

sudo systemctl status mysqld
Note — the service name differs by distribution Ubuntu/Debian call the unit mysql; Fedora/RHEL call it mysqld (the trailing d means "daemon"). If one name reports "could not be found", simply try the other. systemctl list-units | grep -i mysql will tell you which one exists on your machine.

Look for Active: active (running) in the output. inactive (dead) means the server is stopped; failed means it crashed at startup — see Troubleshooting.

4. Starting, stopping and restarting MySQL

When to use them: start after a stop or fresh install, stop before maintenance such as moving data files, restart after changing configuration files. 

sudo systemctl start mysql
sudo systemctl stop mysql
sudo systemctl restart mysql

(Use mysqld instead of mysql on Fedora/RHEL.)

Warning — stopping a production server systemctl stop disconnects every application using the database. On a real system, schedule it and announce it. On your learning machine, stop and start as much as you like.
Tip — start at boot sudo systemctl enable mysql makes the service start automatically on boot; disable reverses it.

5. Securing the MySQL installation

What it does: runs an interactive script that locks down the defaults of a fresh installation. When to use it: once, right after installing — before you create any databases. 

sudo mysql_secure_installation

The script walks you through a series of yes/no questions. It typically asks you to:

  • Set a root password (and optionally enable a password-strength policy).
  • Remove anonymous users — accounts that allow logging in with no username.
  • Disable remote root login — root should only connect from the local machine.
  • Remove the test database — a world-accessible demo database.
  • Reload privilege tables — applies all the changes immediately.
Tip — answer "yes" to everything On a learning or development machine there is no downside to accepting every hardening option. Use a placeholder like strong_password in notes and scripts — never write a real password into a file you might commit to Git.

6. Connecting to MySQL

On Ubuntu, the root database account authenticates via auth_socket by default: any process owned by the Linux root user may connect without a password. That is why the simplest first connection is: 

sudo mysql

Once a root password is set (for example by mysql_secure_installation), or for any password-based account, connect with: 

mysql -u root -p

-u root selects the user, and -p makes the client prompt for the password interactively. A successful login lands you at the mysql> prompt. Type exit or press Ctrl+D to leave.

Warning — never put the password on the command line mysql -u root -pMyPassword works, but the password ends up in your shell history and in the process list where other users can see it. Always use a bare -p and type the password at the prompt.

7. Common MySQL commands

Run these at the mysql> prompt. Every statement ends with a semicolon.

List all databases

SHOW DATABASES;

You'll see system databases such as mysql, information_schema and performance_schema — leave those alone.

Create a database

CREATE DATABASE sample_db;

Switch to a database

USE sample_db;

After USE, table commands apply to sample_db without needing a prefix.

List tables in the current database

SHOW TABLES;

A fresh database returns Empty set — you haven't created any tables yet. See Running SQL files to load a schema.

Delete a database

DROP DATABASE sample_db;
Danger — DROP DATABASE is permanent DROP DATABASE deletes the database and every table and row in it, immediately and without confirmation. There is no undo and no recycle bin. On anything but a throwaway database, take a backup first and double-check the name you typed.

8. MariaDB — the other MySQL

MariaDB is a community fork of MySQL, created in 2009 by MySQL's original developers after Oracle acquired MySQL. It started as a drop-in replacement, and for everything taught on this site the two behave the same: same SQL, same mysql client, same port 3306, same mysqldump-style tooling.

You will meet MariaDB whether you choose it or not, because many Linux distributions ship it as their default "MySQL": plain Debian's default-mysql-server package installs MariaDB, and RHEL-family distributions (RHEL, Rocky, AlmaLinux) carry MariaDB in their standard repositories. Installing it explicitly looks like this: 

# Ubuntu / Debian
sudo apt install mariadb-server

# Fedora / RHEL-based
sudo dnf install mariadb-server

The systemd service is named mariadb:

sudo systemctl status mariadb
sudo systemctl enable --now mariadb

Not sure which one a machine is actually running? Ask the server:

SELECT VERSION();

A MariaDB result contains the word MariaDB (for example 11.4.2-MariaDB); a MySQL result is just a version number like 8.0.39. From the shell, mysql --version tells you the same about the client.

Note — they are slowly drifting apart Since the fork, the two projects have developed independently: recent MySQL and MariaDB versions differ in newer features (JSON handling, authentication plugins, replication options), and MariaDB versions its releases differently (10.x/11.x rather than 8.x). Everything on this site sticks to the shared, compatible core — but if you use advanced features, check the documentation of the one you actually run.
Tip — securing MariaDB MariaDB ships the same hardening script, named mariadb-secure-installation (the old mysql_secure_installation name still works as an alias). The questions are the same as in section 5.
