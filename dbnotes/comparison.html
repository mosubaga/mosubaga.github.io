MySQL vs PostgreSQL

The two databases solve the same problem with different personalities. Use this page as a translation table when switching between them, and as a decision aid when you get to choose.

MySQL2. When to choose MySQL

  • Classic web stacks. LAMP-style apps, WordPress and most PHP CMSes assume MySQL; hosting support is universal.
  • Read-heavy workloads. Mature, simple replication makes scaling reads with replicas straightforward.
  • Team familiarity. If your team and tooling already know MySQL, the operational knowledge is worth a lot.
  • Simple requirements. Plain CRUD with no exotic data types — MySQL is easy to run and hard to misconfigure.

PostgreSQL3. When to choose PostgreSQL

  • Complex queries and analytics. Strong optimizer, window functions, CTEs and full-text search out of the box.
  • Data integrity first. Strict typing and validation by default — bad data is rejected, not silently adjusted.
  • Rich data types. Arrays, ranges, UUID, INET, and best-in-class JSONB for semi-structured data.
  • Extensibility. PostGIS (geodata), TimescaleDB (time series), pgvector (embeddings) turn it into a platform.
  • Standards compliance. Skills and SQL written for Postgres transfer most cleanly to other databases.
Tip — you can't go badly wrong Both are excellent, actively developed and free. For a personal project, pick the one your framework documents first — the experience transfers either way.

4. Important syntax differences

These are the differences most likely to break a script you move between systems.

Identifier quoting and case

MySQL quotes table/column names with backticks; PostgreSQL uses standard double quotes and folds unquoted identifiers to lowercase. 

-- MySQL
SELECT `name` FROM `users`;

-- PostgreSQL
SELECT "name" FROM "users";   -- or simply: SELECT name FROM users;

Strings vs double quotes

In PostgreSQL, double quotes are only for identifiers — string values must use single quotes. MySQL historically tolerated "text" as a string, which masks bugs. Use single quotes for strings everywhere.

Case-insensitive matching

MySQL's default collations compare strings case-insensitively (WHERE name = 'alice' matches Alice); PostgreSQL is case-sensitive and offers ILIKE for insensitive pattern matching.

Other frequent trip-ups

  • ENGINE=InnoDB clauses are MySQL-only — delete them for PostgreSQL.
  • MySQL's LIMIT 10, 20 offset shorthand isn't portable — use LIMIT 20 OFFSET 10, which both accept.
  • PostgreSQL supports RETURNING * on INSERT/UPDATE/DELETE; MySQL (mostly) does not.
  • Strictness: PostgreSQL rejects out-of-range or truncated values with an error; MySQL's behaviour depends on sql_mode (strict by default in modern versions).

6. Auto-increment differences

The classic portability problem: generating sequential primary keys.

-- MySQL
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100)
);

-- PostgreSQL (traditional)
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

-- PostgreSQL (modern, SQL standard)
CREATE TABLE users (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name VARCHAR(100)
);

Under the hood, PostgreSQL's SERIAL creates a separate sequence object that feeds the column its next value; MySQL's AUTO_INCREMENT is a property of the table itself. The effect is the same: insert a row without an id, get the next number.

7. JSON support overview

Both databases can store and query JSON documents inside a relational table — handy for flexible attributes alongside structured columns.

  • MySQL has a JSON type that validates documents on insert, with functions like JSON_EXTRACT() (and the -> / ->> operators) for querying.
  • PostgreSQL has JSON (stored as text) and JSONB (binary, the one you almost always want). JSONB supports containment queries (@>), rich path operators and — crucially — GIN indexes, which make searching inside documents fast. 
-- PostgreSQL: find rows whose JSONB column contains a key/value
SELECT * FROM events WHERE payload @> '{"type": "login"}';

-- MySQL: extract a value from a JSON column
SELECT payload->>'$.type' FROM events;

For JSON-heavy workloads PostgreSQL's JSONB is the stronger tool; for occasional JSON columns either works fine.
