Backup & Restore

Both databases ship a "dump" tool that writes a database out as a plain SQL script — which means a backup is just a file, and a restore is just running a SQL file.

MySQL1. MySQL backup with mysqldump

What it does: mysqldump connects to the server, reads the database, and prints SQL statements that would recreate it (CREATE TABLE + INSERT rows). The > redirects that output into a file. When to use it: before any risky change, before upgrades, and on a regular schedule. 

mysqldump -u username -p database_name > backup.sql

Concrete example:

mysqldump -u root -p sample_db > sample_db_backup.sql

The result is a readable text file — open it with less backup.sql to see exactly what was saved. To dump every database at once, use --all-databases instead of a database name.

MySQL2. MySQL restore

What it does: replays the dump file against a database — identical to running any SQL file. The target database must already exist. 

mysql -u username -p database_name < backup.sql

Restoring into a fresh database:

mysql -u root -p -e "CREATE DATABASE sample_db;"
mysql -u root -p sample_db < sample_db_backup.sql
Warning — restoring overwrites data Dump files usually contain DROP TABLE IF EXISTS statements: restoring into a database replaces its tables with the backup's contents. Triple-check the target database name before pressing Enter.

PostgreSQL3. PostgreSQL backup with pg_dump

What it does: the same idea as mysqldump — writes the database out as SQL text. 

pg_dump -U username database_name > backup.sql

On a default peer-auth setup, run it as the postgres user:

sudo -u postgres pg_dump sample_db > /tmp/sample_db_backup.sql
Tip — other formats and whole-cluster dumps pg_dump -Fc produces a compressed custom-format archive restored with pg_restore (supports parallel and selective restore). pg_dumpall dumps every database plus roles — useful before server migrations.

PostgreSQL4. PostgreSQL restore

What it does: a plain-text dump is restored by running it with psql, like any SQL file. 

psql -U username -d database_name -f backup.sql

Restoring into a fresh database as the postgres user:

sudo -u postgres psql -c "CREATE DATABASE sample_db;"
sudo -u postgres psql -d sample_db -f /tmp/sample_db_backup.sql

Add -v ON_ERROR_STOP=1 to abort on the first error instead of ploughing on through a half-broken restore.

5. Best practices

  • Test restores, not just backups. A backup you have never restored is a hope, not a backup. Periodically restore into a scratch database and check the row counts: sudo -u postgres psql -c "CREATE DATABASE restore_test;" then restore into restore_test and inspect it.
  • Keep backups off the server. If the disk dies, backups stored on the same machine die with the data. Copy dumps to another machine or object storage, e.g. scp backup.sql you@backuphost:/backups/.
  • Never commit backups with real data to GitHub. A dump is your entire database in plain text — customer emails included. Add *.sql.gz and *backup* patterns for dump files to .gitignore (schema-only scripts you wrote by hand are fine to commit).
  • Protect files containing passwords or sensitive data. Restrict dump files (chmod 600 backup.sql) and encrypt them before they leave the machine. The same goes for config files like ~/.my.cnf or ~/.pgpass that store credentials.
  • Automate with cron only after testing manually. First make the dump command work by hand, including the password handling. Then schedule it, e.g. nightly at 02:00: 
    # crontab -e  (illustrative; test the command manually first)
0 2 * * * pg_dump -U appuser sample_db | gzip > /backups/sample_db_$(date +\%F).sql.gz
    Check for a fresh file the next morning — a silently failing cron backup is worse than no backup, because you stop worrying.
