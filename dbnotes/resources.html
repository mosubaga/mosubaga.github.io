5. Basic SQL crash course

The statements below work identically in MySQL and PostgreSQL — this is the portable core of SQL. Run them in either client against a test database.

CREATE TABLE — define structure

CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(10, 2),
    stock INT DEFAULT 0
);

INSERT — add rows

INSERT INTO products (id, name, price, stock)
VALUES (1, 'Keyboard', 49.90, 12),
       (2, 'Mouse', 19.90, 30),
       (3, 'Monitor', 199.00, 5);

SELECT + WHERE — read rows with conditions

SELECT name, price FROM products
WHERE price < 100;

ORDER BY — sort results

SELECT name, price FROM products
ORDER BY price DESC;

UPDATE — change rows

UPDATE products
SET stock = stock - 1
WHERE id = 2;

DELETE — remove rows

DELETE FROM products
WHERE id = 3;
Warning — UPDATE and DELETE without WHERE Leaving off the WHERE clause applies the change to every row in the table. Habit worth building: write the WHERE first as a SELECT, check what comes back, then turn it into the UPDATE/DELETE.

JOIN — combine tables

SELECT orders.id, customers.name
FROM orders
JOIN customers ON orders.customer_id = customers.id;

GROUP BY — aggregate

SELECT customer_id, COUNT(*) AS order_count
FROM orders
GROUP BY customer_id;

6. Sample database project

A small web-shop schema to practise everything on this site: save it as shop.sql, create a database, and run the file in either system. Only the primary-key line differs between the two — pick the right variant for your database. 

-- shop.sql — works in MySQL and PostgreSQL
-- For auto-generated ids, replace "INT PRIMARY KEY" with:
--   MySQL:       INT AUTO_INCREMENT PRIMARY KEY
--   PostgreSQL:  SERIAL PRIMARY KEY

CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);

CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(10, 2) NOT NULL
);

CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT NOT NULL REFERENCES customers(id),
    order_date DATE NOT NULL
);

CREATE TABLE order_items (
    id INT PRIMARY KEY,
    order_id INT NOT NULL REFERENCES orders(id),
    product_id INT NOT NULL REFERENCES products(id),
    quantity INT NOT NULL DEFAULT 1
);

INSERT INTO customers VALUES (1, 'Alice', 'alice@example.com'),
                             (2, 'Bob', 'bob@example.com');
INSERT INTO products  VALUES (1, 'Keyboard', 49.90),
                             (2, 'Mouse', 19.90),
                             (3, 'Monitor', 199.00);
INSERT INTO orders    VALUES (1, 1, '2026-06-01'),
                             (2, 2, '2026-06-03');
INSERT INTO order_items VALUES (1, 1, 1, 1),
                               (2, 1, 2, 2),
                               (3, 2, 3, 1);

Then try a query that touches every table:

SELECT customers.name,
       products.name AS product,
       order_items.quantity
FROM order_items
JOIN orders    ON order_items.order_id = orders.id
JOIN customers ON orders.customer_id = customers.id
JOIN products  ON order_items.product_id = products.id
ORDER BY customers.name;
Tip — practise the full cycle With this schema you can practise the whole learning path: create a database, run the file, create an appuser with access to it, back it up, drop it, and restore it.

7. Connecting applications to databases

Applications usually connect with a connection string (or URL) that packs user, password, host, port and database into one line: 

mysql://appuser:password@localhost:3306/sample_db
postgresql://appuser:password@localhost:5432/sample_db

The exact format depends on the language and library — some take a URL like the above, others take separate host/user/password parameters. Whatever the format: use your limited application user (never root/postgres), and keep the string out of source code — load it from an environment variable such as DATABASE_URL.

Common client libraries to look up for your language:

  • Python — mysql-connector-python / psycopg (or SQLAlchemy on top).
  • Go — go-sql-driver/mysql / jackc/pgx.
  • JavaScript (Node.js) — mysql2 / pg (or Prisma, Drizzle).
  • Java — JDBC drivers: MySQL Connector/J / PostgreSQL JDBC.

8. The Docker option

Both databases also run as Docker containers — handy for throwaway test instances (docker run -e POSTGRES_PASSWORD=strong_password -p 5432:5432 postgres). Everything you learned here still applies inside the container: same clients, same SQL, same users and dumps. This site focuses on direct Linux installation because understanding the service, its files and its authentication makes the Docker variant trivial later — not the other way around.

9. Suggested next topics

Roughly in the order they pay off:

  • Indexes — the single biggest lever for query speed; learn CREATE INDEX and when it helps.
  • Joins — go beyond INNER JOIN: LEFT/RIGHT joins and when each returns NULLs.
  • Transactions — BEGIN/COMMIT/ROLLBACK and why "all or nothing" matters.
  • Views — save complex queries as virtual tables.
  • Stored procedures — logic that runs inside the database.
  • Triggers — automatic actions on INSERT/UPDATE/DELETE.
  • Database normalization — designing tables that avoid duplicated data.
  • Query performance — reading EXPLAIN output to see what the database actually does.
  • Docker-based database setup — reproducible test databases in CI pipelines.
  • Connecting from code — Python, Go, JavaScript, Java… see the libraries above.
