User Management & Permissions

Applications should never connect as root or postgres. This page shows how to create limited users in both databases, grant only the privileges they need, and clean up users you no longer want.

Placeholder passwords Every example uses strong_password as a stand-in. Replace it with a long, randomly generated password — and never commit a real one to a Git repository.

MySQL1. MySQL user management

Create a user

What it does: creates an account that may connect from a specific host. When to use it: one user per application or service. 

CREATE USER 'appuser'@'localhost' IDENTIFIED BY 'strong_password';

Understanding 'user'@'host'

A MySQL account is the combination of a name and the host it may connect from — 'appuser'@'localhost' and 'appuser'@'%' are two different accounts with separate passwords and privileges.

  • 'appuser'@'localhost' — may connect only from the same machine.
  • 'appuser'@'192.168.1.50' — may connect only from that IP address.
  • 'appuser'@'%' — may connect from anywhere.
Warning — avoid '%' unless you need it A '%' account is reachable from any machine that can see the port. Prefer localhost, or the specific IP of your application server. This is also why mysterious "Access denied" errors happen: the user exists for localhost but the connection comes from elsewhere — see Troubleshooting.

Grant privileges

What it does: gives the user rights on a scope — here, everything inside sample_db (the .* means "all tables in that database"). 

GRANT ALL PRIVILEGES ON sample_db.* TO 'appuser'@'localhost';
FLUSH PRIVILEGES;

FLUSH PRIVILEGES; reloads the grant tables. Strictly speaking, modern MySQL applies GRANT statements immediately and the flush is only required after editing grant tables directly — but including it costs nothing and is a widespread habit.

Tip — grant less than ALL A typical web application only needs row access, not the right to drop tables: GRANT SELECT, INSERT, UPDATE, DELETE ON sample_db.* TO 'appuser'@'localhost';

List users

SELECT user, host FROM mysql.user;

To inspect one account's rights:

SHOW GRANTS FOR 'appuser'@'localhost';

Remove a user

DROP USER 'appuser'@'localhost';
Danger — DROP USER is immediate The account and its grants are removed permanently. Any application still using those credentials starts failing on its next connection. Existing open sessions, however, stay alive until they disconnect.

Why applications must not use root

root can read and drop every database on the server. If an attacker exploits your application (for example via SQL injection), they inherit the application's database privileges. With a limited appuser the damage is contained to one database; with root it is total. The same logic says remote root login should stay disabled — administer locally or over SSH.

PostgreSQL2. PostgreSQL user and role management

Roles: the one concept to understand

PostgreSQL has a single concept, the role, which covers what other systems split into "users" and "groups". A role with the LOGIN attribute can connect — that's what we informally call a user. A role without LOGIN acts as a group: you grant privileges to it once, then grant the group role to individual login roles. CREATE USER is simply shorthand for CREATE ROLE … LOGIN.

Create a user (login role)

CREATE USER appuser WITH PASSWORD 'strong_password';

Note the difference from MySQL: no @host part. In PostgreSQL, where connections may come from is controlled separately, in the pg_hba.conf file — roles themselves are host-independent.

Create a database owned by the user

What it does: creates the database and makes appuser its owner. When to use it: the cleanest setup for a dedicated application database — the owner automatically has full rights inside it, including creating tables. 

CREATE DATABASE sample_db OWNER appuser;

Grant privileges on an existing database

GRANT ALL PRIVILEGES ON DATABASE sample_db TO appuser;
Note — database privileges ≠ table privileges This grant covers database-level rights (connecting, creating schemas). It does not grant access to tables owned by someone else — a frequent surprise. Tables live inside schemas (the default one is public), and table access is granted separately, e.g.: GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO appuser; Making appuser the database owner (above) sidesteps the whole issue for simple projects.

List roles

Inside psql:

\du

The output shows each role's attributes (Superuser, Create DB, …) — login roles and group roles side by side.

Remove a user

DROP USER appuser;
Danger — DROP USER and owned objects The removal is permanent, and PostgreSQL refuses to drop a role that still owns objects (databases, tables). Reassign them first with REASSIGN OWNED BY appuser TO postgres; — and make sure nothing in production still connects as that role.

Why limited users matter (again)

The postgres role is a superuser: it bypasses every permission check. Applications should connect as a role that owns (or has limited rights on) exactly one database — nothing else. Group roles make this scalable: create a readonly role once, grant it SELECT everywhere it needs, then grant readonly to each analyst's login role.

3. Database security checklist

These rules apply equally to MySQL and PostgreSQL — most real-world database breaches violate one of them.

  • Never use admin accounts in applications. root and postgres are for administration only. Every app gets its own limited user.
  • Use strong, unique passwords. Long and randomly generated, one per account, stored in a password manager or secret store — not reused from other systems.
  • Grant only what is needed (least privilege). Most applications need SELECT, INSERT, UPDATE, DELETE on one database. Grant ALL only when you can say why.
  • Do not expose database ports publicly. MySQL (3306) and PostgreSQL (5432) should listen on localhost or a private network. Both default to local-only listening — think twice before changing that.
  • Use a firewall. Even on a private network, restrict the port to the hosts that need it (ufw, firewalld — see Troubleshooting).
  • Remote administration: SSH tunnels or VPN. Instead of opening the port, tunnel it: ssh -L 3306:localhost:3306 you@dbserver then connect to localhost:3306 on your machine. Same pattern for 5432.
  • Keep real passwords out of Git. Use environment variables or a secrets manager for connection credentials. If a password ever lands in a public repository, treat it as compromised and rotate it immediately — deleting the commit is not enough.
  • Treat backup files as sensitive. A dump file contains all your data, including user tables. Encrypt backups, restrict file permissions, and never commit dumps with real data — see Backup & restore.
