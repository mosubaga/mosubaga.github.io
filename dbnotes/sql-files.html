Running SQL Files

A .sql file lets you build a schema or load data with one command instead of typing statements by hand. Here is how to run scripts in both databases — from the shell and from inside the client.

1. What is a SQL file?

A SQL file is a plain text file (conventionally ending in .sql) containing one or more SQL statements, each terminated by a semicolon. The database executes them top to bottom, exactly as if you had typed them at the prompt. A typical small script: 

CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

INSERT INTO users VALUES (1, 'Alice', 'alice@example.com');

SQL files are how teams share schemas, seed test data, apply migrations and restore backups — so being able to run one reliably is a core skill.

Tip — keep schema files in version control Schema scripts belong in Git next to your code. Backup dumps with real data do not — see Backup & restore.

MySQL2. Running a SQL file in MySQL

What it does: feeds the file to the mysql client via shell input redirection (<). The client runs every statement against the named database, then exits. 

mysql -u username -p database_name < file.sql

Concrete example — load schema.sql into sample_db as root:

mysql -u root -p sample_db < schema.sql

You are prompted for the password, the script runs, and you get your shell prompt back. No output means success — MySQL only prints errors. The database (sample_db here) must already exist; the script runs inside it.

PostgreSQL3. Running a SQL file in PostgreSQL

What it does: psql has a dedicated flag, -f, that reads and executes a file. -d names the target database. 

psql -U username -d database_name -f file.sql

On a default Linux setup with peer authentication, run it as the postgres system user: 

sudo -u postgres psql -d sample_db -f schema.sql

Unlike MySQL, psql prints a confirmation tag for every statement (CREATE TABLE, INSERT 0 1, …), so you can see exactly how far the script got.

Tip — stop on the first error By default psql keeps going after an error. Add -v ON_ERROR_STOP=1 to abort at the first failure — almost always what you want for schema scripts.

4. Running SQL files interactively

When to use it: when you are already inside the client — exploring, then deciding to load a script without leaving your session.

MySQLWith SOURCE

Connect first:

mysql -u root -p

Then, at the mysql> prompt:

USE sample_db;
SOURCE schema.sql;

PostgreSQLWith \i

Connect first:

psql -U username -d database_name

Then, at the psql prompt:

\i schema.sql
Note — relative paths In both clients, a relative filename like schema.sql is resolved against the directory you launched the client from — not your home directory. When in doubt, use an absolute path such as /home/you/sql/schema.sql.

5. Common problems when running SQL files

File not found

The path is wrong or relative to a different directory than you think. Check with ls -l file.sql from the same shell, or use an absolute path. With sudo -u postgres, remember the command runs as another user — but the path is still resolved from your current directory, so a file in a directory the postgres user cannot enter leads to the next problem.

Permission denied

The user running the client cannot read the file. Classic case: sudo -u postgres psql -f /home/you/schema.sql fails because the postgres user cannot read your home directory. Fix it by copying the file to a shared location first: 

cp schema.sql /tmp/
sudo -u postgres psql -d sample_db -f /tmp/schema.sql

Database does not exist

Running a file does not create the target database. Create it first (CREATE DATABASE sample_db;), then run the script against it. The error looks like ERROR 1049 (42000): Unknown database in MySQL or FATAL: database "sample_db" does not exist in PostgreSQL.

Syntax differences between MySQL and PostgreSQL

A script written for one database often fails on the other. The usual culprits: AUTO_INCREMENT (MySQL) vs SERIAL/identity columns (PostgreSQL), backtick-quoted identifiers (`name`, MySQL-only) vs double quotes (standard), and engine clauses like ENGINE=InnoDB. See syntax differences for the full list.

Character encoding issues

If accented characters or emoji come out garbled, the file's encoding doesn't match what the server expects. Save files as UTF-8 (check with file file.sql) and prefer UTF-8 databases: utf8mb4 in MySQL, UTF8 in PostgreSQL.

Running the file against the wrong database

Warning — the quiet disaster The most dangerous failure is the one with no error message: the script runs perfectly, but against the wrong database (for example, production instead of test). Before running any script that drops or modifies tables, confirm the target: SELECT DATABASE(); in MySQL, SELECT current_database(); or \conninfo in psql.
