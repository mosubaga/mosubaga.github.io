Troubleshooting

Most database problems on Linux fall into five buckets: the service isn't running, authentication is rejecting you, the port is taken, or a firewall is in the way. Work through them in that order.

1. Service will not start

First rule of debugging: read what the system is actually telling you. Start with the service status — it shows the state and the last few log lines: 

sudo systemctl status mysql
sudo systemctl status postgresql

If the status output isn't enough, ask the journal for recent errors with full detail:

journalctl -xe

Or follow one service's log specifically:

sudo journalctl -u mysql --since "10 minutes ago"
sudo journalctl -u postgresql --since "10 minutes ago"

Typical causes, in rough order of likelihood:

  • Wrong service name — mysql vs mysqld (see MySQL page).
  • PostgreSQL data directory not initialised on Fedora/RHEL — run sudo postgresql-setup --initdb (see PostgreSQL page).
  • Config file typo — a single bad line in my.cnf or postgresql.conf stops startup; the log names the line.
  • Disk full — check with df -h; databases refuse to start without space.
  • Port already in use — see port conflicts below.
Tip — copy the exact error into a search engine Database error messages are precise and well indexed. The full line from the log almost always leads to the answer faster than guessing.

MySQL2. Access denied in MySQL

The classic error:

ERROR 1045 (28000): Access denied for user 'appuser'@'localhost' (using password: YES)

Note that the message tells you which account MySQL resolved you to — the name and the host. The usual causes:

  • Wrong password. The boring one — check for typos and stray whitespace, and remember passwords are case-sensitive.
  • Wrong host. MySQL accounts are 'user'@'host' pairs. If the user exists as 'appuser'@'localhost' but you connect over the network (even from the same machine via its IP), there is no matching account. Check what exists: 
    SELECT user, host FROM mysql.user;
  • Privileges never granted. The user can log in but gets "Access denied" on a specific database — verify with SHOW GRANTS FOR 'appuser'@'localhost'; and grant what is missing.
  • Forgot FLUSH PRIVILEGES. Only relevant if grant tables were edited directly with INSERT/UPDATE — but if you suspect stale privileges, running FLUSH PRIVILEGES; is harmless.
  • Trying mysql -u root -p on a fresh Ubuntu install. Root may use socket authentication with no password — use sudo mysql instead (see connecting).

PostgreSQL3. PostgreSQL: peer authentication failed

The most-searched PostgreSQL error on Linux:

FATAL: Peer authentication failed for user "postgres"

What it means: for local connections PostgreSQL defaults to peer authentication — instead of asking for a password, it checks that your Linux username matches the database role you are asking for. Running psql -U postgres as your own user fails, because you are not the Linux user postgres.

Fix 1 — work with peer auth (recommended)

Become the matching Linux user for admin work; no configuration changes needed:

sudo -u postgres psql

Fix 2 — connect over TCP with a password

Password-based rules usually apply to TCP connections, so forcing a network connection to localhost prompts for a password instead: 

psql -h 127.0.0.1 -U appuser -d sample_db

(The role needs a password set: ALTER USER appuser WITH PASSWORD 'strong_password';)

Fix 3 — change the authentication rules

Authentication behaviour lives in pg_hba.conf ("host-based authentication"); the data directory location and other server settings live in postgresql.conf. Find them with: 

sudo -u postgres psql -c "SHOW hba_file;"
sudo -u postgres psql -c "SHOW config_file;"

Changing the local rule from peer to scram-sha-256 makes local connections ask for a password. After editing, apply with sudo systemctl reload postgresql.

Warning — edit pg_hba.conf carefully A wrong line here can lock everyone out, including you, or accidentally open the server to passwordless logins (trust). Before editing: make a copy (sudo cp pg_hba.conf pg_hba.conf.bak), change one line at a time, keep a working sudo -u postgres psql session open while you test, and never use trust outside a disposable lab machine.

4. Port conflicts

The default ports:

MySQL:      3306
PostgreSQL: 5432

When to check: the service fails to start with an "address already in use" error, or you suspect another instance (a Docker container is the usual suspect) is occupying the port. ss shows who is listening: 

ss -tulpn | grep 3306
ss -tulpn | grep 5432

The flags: tcp, udp, listening, processes, numeric. The last column names the process that owns the port (run with sudo to see processes you don't own). If something else is on the port, either stop it or move one of the two to a different port in its config file.

5. Firewall issues

Symptom: local connections work, but a remote client hangs or times out. On Ubuntu, check the firewall state: 

sudo ufw status

(Fedora/RHEL use firewalld: sudo firewall-cmd --list-all.) If the database port isn't allowed, remote packets are dropped silently — which is exactly the timeout you're seeing.

Warning — do not open database ports to the internet Publicly reachable MySQL/PostgreSQL ports are scanned and attacked within minutes of going online. If a remote machine needs access, allow only its IP (e.g. sudo ufw allow from 192.168.1.50 to any port 5432) — and prefer an SSH tunnel or VPN instead, as described in the security checklist. Also remember the server must be configured to listen beyond localhost at all (bind-address in MySQL, listen_addresses in PostgreSQL) — the safe default is local-only.
