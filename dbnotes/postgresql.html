PostgreSQLPostgreSQL on Linux

Install PostgreSQL from your distribution's packages, initialise the data directory where needed, control the service with systemd, and learn psql and its meta-commands.

1. Installing PostgreSQL on Ubuntu / Debian

What it does: installs the PostgreSQL server, client tools, and the postgresql-contrib package of useful extensions. When to use it: on any Debian-family distribution. 

sudo apt update
sudo apt install postgresql postgresql-contrib

On Ubuntu/Debian the installer automatically creates the database cluster (the on-disk data directory) and starts the service, so PostgreSQL is ready to use immediately after this command finishes.

Tip — what is "contrib"? postgresql-contrib adds officially maintained extensions such as pgcrypto and hstore. It is small and worth installing from the start.

2. Installing PostgreSQL on Fedora / RHEL-based systems

When to use it: on Fedora, RHEL, CentOS Stream, Rocky Linux or AlmaLinux. Note the extra step: on these systems you must initialise the database cluster yourself before the server can start. 

sudo dnf install postgresql-server postgresql-contrib
sudo postgresql-setup --initdb

postgresql-setup --initdb creates the data directory (usually /var/lib/pgsql/data) with the system files PostgreSQL needs. Then start and enable the service: 

sudo systemctl enable --now postgresql
Note — skipping initdb is the #1 first-run error If you forget --initdb, systemctl start postgresql fails with an error about a missing or invalid data directory. Ubuntu users never see this because their packages run it automatically.

3. Checking the PostgreSQL service status

When to use it: after installation and whenever a connection is refused — confirm the server is actually running before debugging anything else. 

sudo systemctl status postgresql

Look for Active: active (running). Unlike MySQL, the unit is called postgresql on virtually every distribution, so there is no service-name guessing game here.

4. Starting, stopping and restarting PostgreSQL

When to use them: the same as for any systemd service — restart after editing postgresql.conf or pg_hba.conf, stop before file-level maintenance. 

sudo systemctl start postgresql
sudo systemctl stop postgresql
sudo systemctl restart postgresql
Tip — reload instead of restart Many configuration changes (including most pg_hba.conf edits) only need sudo systemctl reload postgresql, which re-reads the config without dropping client connections.

5. Connecting to PostgreSQL

The PostgreSQL packages create a Linux system user named postgres, and inside the database a superuser role with the same name. By default, local connections use peer authentication: you are allowed in as database user postgres only if you are the Linux user postgres. That is why the standard first connection looks like this: 

sudo -u postgres psql

sudo -u postgres runs the command as the postgres Linux user, and psql opens an interactive session. You will land at the postgres=# prompt. Leave with \q.

Note — why psql -U postgres alone often fails Running psql -U postgres as your own Linux user typically produces FATAL: Peer authentication failed — peer auth checks your Linux username, not a password. See Troubleshooting for the full explanation and fixes.

6. Common PostgreSQL commands

psql understands two different kinds of input, and telling them apart is the key to feeling at home:

  • Meta-commands start with a backslash (\l, \dt, …). They are shortcuts built into the psql client, not SQL. They need no semicolon and will not work from any other tool.
  • SQL statements (CREATE DATABASE …;) are sent to the server, end with a semicolon, and work from any client or programming language.

Essential meta-commands

\l            -- list all databases
\c sample_db  -- connect (switch) to database sample_db
\dt           -- list tables in the current database
\du           -- list users/roles
\q            -- quit psql
Tip — get help inside psql \? lists every meta-command, and \h CREATE TABLE shows the SQL syntax for any statement.

Creating and dropping databases (regular SQL)

CREATE DATABASE sample_db;
DROP DATABASE sample_db;
Danger — DROP DATABASE is permanent Exactly as in MySQL, DROP DATABASE destroys every table and row instantly and irreversibly. PostgreSQL also refuses to drop the database you are currently connected to — switch away with \c postgres first. Back up anything you care about before dropping.
